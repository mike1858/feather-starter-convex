# Phase 999.4: Integration Gaps — Research

**Completed:** 2026-03-29
**Mode:** Implementation research (ecosystem + codebase analysis)

## Current State Assessment

### Two Parallel Code Generation Systems

The codebase has **two independent generation systems** that were never unified:

1. **Plop-based generators** (`generators/*.js` + `templates/feature/*.hbs`) — Production system. Used for all CalmDo features (tasks, projects, subtasks, work-logs, activity-logs). Outputs to `src/features/` and `convex/`. Has 26 Handlebars templates, auto-wiring via string manipulation (`generators/utils/wiring.js`), interactive wizard. **This is what `npm run gen:feature` runs.**

2. **Pipeline system** (`templates/pipeline/` + `templates/wiring/` + `bin/feather.ts`) — Architecture prototype. Built during Phase 999.1. Uses ts-morph AST wiring, outputs to `src/generated/` and `convex/generated/`, has stub file generation (not real templates). **This is what `npx feather generate` would run.** Never used for real features.

### What Each System Has

| Capability | Plop Generators | Pipeline System |
|-----------|----------------|-----------------|
| Template rendering | 26 Handlebars templates (full CRUD) | Inline stub generators (5 minimal stubs) |
| Auto-wiring | String manipulation in wiring.js | ts-morph AST manipulation in wiring/*.ts |
| Output targeting | `src/features/` only (legacy) | `src/generated/` + `convex/generated/` |
| CLI interface | `npm run gen:feature` (Plop prompts) | `npx feather generate` (Commander.js) |
| YAML validation | yaml-resolver.js (simple merge) | Zod schema validation + defaults merger |
| Cross-entity | Not supported | resolver.ts exists (no templates) |
| Update command | Not supported | `feather update` exists (uses pipeline stubs) |
| Tests | Generator __tests__ (output verification) | pipeline.test.ts (integration), wiring/*.test.ts |

### The Integration Gap

The pipeline system has the right architecture (generated/custom split, AST wiring, CLI, update command, Zod validation) but produces **stubs, not real code**. The Plop system produces **real, working code** but targets the wrong directory and uses fragile string manipulation for wiring.

## Architecture Patterns

### Pattern 1: Pipeline as Orchestrator, Plop Templates as Renderers

The `generateFeature()` function in `templates/pipeline/generate.ts` should become the single entry point. Instead of its current stub generators, it should invoke Plop's template rendering (Handlebars) for file content generation.

```
generateFeature(options)
  → 1. Read & validate YAML (Zod schema — already works)
  → 2. Merge with defaults (defaults-merger — already works)
  → 3. Render templates (Handlebars — steal from Plop generators)
  → 4. Write to generated/ or features/ (scaffold.ts — already works)
  → 5. Wire shared files (ts-morph wiring — already works)
```

**Key insight:** Plop's `smartAdd` action and Handlebars rendering can be extracted and used without the Plop CLI. The Handlebars helpers in `generators/utils/handlebars-helpers.js` and templates in `templates/feature/*.hbs` are the real assets. The Plop prompts/wizard are a convenience layer on top.

### Pattern 2: Keep Plop for Interactive, Pipeline for Programmatic

Alternative: keep both systems but make Plop call the pipeline internally.

```
npm run gen:feature (interactive)
  → Plop prompts → builds featureConfig
  → calls generateFeature() from pipeline
  → pipeline handles scaffold + wire

npx feather generate (programmatic)
  → reads YAML directly
  → calls generateFeature() from pipeline
```

This pattern is cleaner because Plop's value is its interactive wizard, while the pipeline's value is its programmatic API.

### Pattern 3: Cross-Entity Panel Templates

The resolver (`templates/cross-entity/resolver.ts`) already produces a `DetailPageContext` with typed `ResolvedRelatedRecord[]`. Each related record has a `display` type that maps to a panel template:

| Display Type | Panel Template | CalmDo Usage |
|-------------|---------------|--------------|
| `table` | Related records in a sortable table | project-tasks |
| `checklist` | Inline checkable list with reorder | subtasks |
| `timeline` | Chronological entries with timestamps | work-logs, activity-logs |
| `cards` | Grid of card previews | project overview |
| `related-record` | Not in schema — removed from scope | N/A |

**Note:** The `relatedRecordSchema` in `detail-view.schema.ts` defines `display: z.enum(["table", "checklist", "timeline", "cards"])` — only 4 types, not 5. The CONTEXT.md mentions "related-record" as a 5th type but the Zod schema doesn't include it. The 4 types from the schema should be the scope.

### Pattern 4: CalmDo Migration as Generated/Custom Split Proof

The migration should retarget CalmDo features to prove the generated/custom split works. However, this is a **large operation** that touches 300+ tests and import paths across the entire codebase. The CONTEXT.md says "all-at-once migration" but this should be carefully sequenced:

1. First: make the pipeline produce real code (replace stubs with Handlebars output)
2. Then: test with a fresh entity (not existing CalmDo)
3. Then: migrate CalmDo features to generated/custom split
4. Finally: update all imports and test paths

## Common Pitfalls

### Pitfall 1: Plop Handlebars Helpers Not Available Outside Plop

The generators register custom Handlebars helpers via `generators/utils/handlebars-helpers.js`. These include `pascalCase`, `camelCase`, `kebabCase`, `eq`, `or`, `and`, `ifCond`, `json`, `apiPath`, `errorsPath`, etc. If the pipeline renders Handlebars templates directly, it must register the same helpers. Otherwise templates will fail silently or produce broken output.

**Mitigation:** Import and register all helpers from `handlebars-helpers.js` before rendering any template.

### Pitfall 2: Plop's `smartAdd` vs Direct File Write

Plop's `smartAdd` action (from `generators/utils/smart-add.js`) has special behavior: it checks if the file already exists and handles marked regions for updates. Direct `fs.writeFileSync` would overwrite everything. The `feather update` command specifically wants to overwrite generated files, so this is actually correct for the pipeline — but for first-time generation, we need to be careful not to overwrite custom code.

**Mitigation:** `outputMode: "generated"` targets `src/generated/` which is always overwritable. `outputMode: "legacy"` targets `src/features/` where `smartAdd` behavior matters.

### Pitfall 3: Dual Wiring Systems

Currently there are TWO wiring approaches:
- `generators/utils/wiring.js` — string manipulation (used by Plop)
- `templates/wiring/*.ts` — ts-morph AST (used by pipeline)

Both are idempotent but they produce slightly different output formatting. Running both would double-wire. The plan must pick ONE and retire the other.

**Recommendation:** Use ts-morph wiring (more reliable, handles edge cases like `as const`). Retire string manipulation wiring. This means the Plop generators must also use the pipeline's wiring or call through to it.

### Pitfall 4: CalmDo Migration Scope Creep

Migrating all CalmDo features to generated/custom split will touch:
- 5 feature directories (tasks, projects, subtasks, work-logs, activity-logs)
- `convex/schema.ts` imports
- Route files importing from features
- 300+ test import paths
- `src/shared/nav.ts`, `src/shared/errors.ts`, `src/i18n.ts`

This is a v1.0-scale restructure. It should NOT be mixed with template development or pipeline integration. It deserves its own plan.

### Pitfall 5: `feather update` Deletes Then Regenerates

The current `update.ts` deletes `src/generated/{name}/` and `convex/generated/{name}/` before regenerating. If the pipeline produces stubs instead of real code, running `feather update` would replace working code with stubs. The pipeline must produce full code before `feather update` is safe to use.

### Pitfall 6: package.json `bin` Entry Missing

The CLI entry point `bin/feather.ts` exists but `package.json` has no `bin` field. `npx feather` won't work until this is added. The shebang line uses `#!/usr/bin/env npx tsx` which requires `tsx` to be available.

## Code Examples

### Handlebars Template Rendering Without Plop

```typescript
import Handlebars from "handlebars";
import { registerHelpers } from "../generators/utils/handlebars-helpers";
import * as fs from "node:fs";
import * as path from "node:path";

// Register all custom helpers
registerHelpers(Handlebars);

function renderTemplate(templatePath: string, data: Record<string, unknown>): string {
  const raw = fs.readFileSync(templatePath, "utf-8");
  const template = Handlebars.compile(raw);
  return template(data);
}
```
