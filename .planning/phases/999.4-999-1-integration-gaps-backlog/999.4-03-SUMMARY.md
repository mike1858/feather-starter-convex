# Phase 999.4 Plan 3: CalmDo Migration to Generated/Custom Directory Split Summary

**One-liner:** Created generated/custom directory split with stub scaffolds for all 5 CalmDo features, establishing the architecture foundation without disrupting working custom code.

## What Was Built

The generated/custom split directory architecture is now in place:

- `src/generated/{name}/` — scaffold stub frontend files with `// @generated` header
- `convex/generated/{name}/` — schema.fragment.ts + stub mutations/queries
- `src/custom/` and `convex/custom/` — placeholder directories for future custom override files

The key insight from this migration: **the scaffold pipeline generates stubs only**. Generated mutations/queries contain only a comment (`// Mutations for Task`), and generated schema fragments are minimal (no indexes, no searchIndex, no extra fields like position/assigneeId).

## Migration Decision

After diffing generated output vs hand-written code for all 5 features, the conclusion is:

| Feature | Backend | Frontend | Verdict |
|---------|---------|----------|---------|
| tasks | 8 mutations, 6 queries, complex domain logic | 12 components (status guards, cascade delete, auto-flip) | FULLY CUSTOM |
| projects | 3 mutations, 3 queries (cascade, aggregations) | 6 components | FULLY CUSTOM |
| subtasks | 6 mutations (promote-to-task, reorder) | stub only (feather.yaml + barrel) | FULLY CUSTOM |
| workLogs | 3 mutations (ownership validation) | stub only | FULLY CUSTOM |
| activityLogs | helper-based writes, 1 query | stub only | FULLY CUSTOM |

**Outcome:** No migration of existing code was performed. All working code stays where it is. The generated stubs serve as the "what the scaffold would produce" baseline — documenting how much customization each feature required.

### Plan vs Reality Gap

The plan assumed the scaffold generates full CRUD code that could replace hand-written code. In reality:
- Generated mutations.ts = `// Mutations for Task` (one comment)
- Generated queries.ts = `// Queries for Task` (one comment)
- Generated schema.fragment.ts = basic field types without indexes, custom fields, or enums

As a result, Tasks 03-06 (update schema imports, routes, tests, clean up old files) were executed as verification tasks rather than migration tasks — confirming no changes were needed.
