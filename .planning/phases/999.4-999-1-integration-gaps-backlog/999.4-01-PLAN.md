# Plan 999.4-01: Pipeline Template Integration — Replace Stubs with Handlebars Rendering + CLI Bin Entry

## Objective

Replace the 5 inline stub generators in `templates/pipeline/scaffold.ts` with real Handlebars template rendering using the existing 26 templates in `templates/feature/`. After this plan, `generateFeature()` produces the same quality output as the Plop generators, and `npx feather generate` works as a CLI command.

## Why This Matters

The pipeline system has the right architecture (generated/custom split, AST wiring, CLI, Zod validation) but produces stubs — not real code. The Plop system produces real, working code but targets the wrong directory. This plan connects the pipeline's orchestration to Plop's rendering, making the pipeline the single code generation entry point.

## must_haves

- [ ] `scaffold.ts` renders Handlebars templates instead of inline stubs
- [ ] All 26 `.hbs` templates in `templates/feature/` are renderable through the pipeline
- [ ] Custom Handlebars helpers from `generators/utils/handlebars-helpers.js` are registered (adapted from Plop API to standalone Handlebars)
- [ ] `package.json` has a `bin` entry pointing to `bin/feather.ts`
- [ ] `npx feather generate todos --dry-run` produces correct file listing
- [ ] Pipeline integration tests verify generated file content matches expected patterns
- [ ] `npm test` passes at 100% coverage

<title>Extract Handlebars helper registration from Plop-coupled API to standalone Handlebars</title>
<type>code</type>
<read_first>
- generators/utils/handlebars-helpers.js (current helpers — uses plop.setHelper() API)
- generators/utils/types.ts (FieldConfig type definition used by helpers)
- templates/feature/schema.ts.hbs (template that uses zodType, convexValidatorType helpers)
- templates/feature/form.tsx.hbs (template that uses formInputType, ifEq helpers)
- templates/feature/mutations.ts.hbs (template that uses apiPath, errorsPath helpers)
</read_first>
<action>
Create `templates/pipeline/render.ts` — standalone Handlebars template renderer:

1. Import Handlebars directly (not through Plop): `import Handlebars from "handlebars"`
2. Create `registerStandaloneHelpers(hbs: typeof Handlebars): void` that registers ALL helpers from `handlebars-helpers.js`, adapted from `plop.setHelper(name, fn)` to `hbs.registerHelper(name, fn)`.

   Helpers to register (complete list from handlebars-helpers.js):
   - `zodType` — returns Zod type string for a field config
   - `zodTypeExport` — returns named const export for enum fields
   - `convexValidatorType` — returns Convex validator for schema.ts
   - `formInputType` — returns React component name for form inputs
   - `ifEq` — equality conditional block helper
   - `eq` — equality subexpression helper
   - `ifIn` — array membership conditional
   - `or` — logical OR subexpression
   - `and` — logical AND subexpression
   - `not` — logical NOT subexpression
   - `json` — JSON.stringify helper
   - `enumBadgeColor` — dot/text color pair for enum values
   - `enumColorEntries` — all enum color mappings for a field
   - `showOnView` — visibility check for field in a view context
   - `apiPath` — bracket notation for kebab-case API paths (e.g., `api["work-logs"]`)
   - `errorsPath` — bracket notation for error constant paths
   - `toUpperSnakeCase` — UPPER_SNAKE_CASE conversion

   Also register Plop's built-in helpers that templates may use:
   - `pascalCase` — via `change-case` package (already a Plop dependency)
   - `camelCase`
   - `kebabCase`
   - `snakeCase`
   - `titleCase`
   - `lowerCase`
   - `upperCase`

3. Create `renderTemplate(templatePath: string, data: Record<string, unknown>): string` — reads .hbs file, compiles with Handlebars, renders with data.

4. Create `renderFeatureTemplates(config: FeatureYaml, templateDir: string): Map<string, string>` — takes a validated FeatureYaml config, builds the template data context (matching what Plop generators build in plopfile.js), and renders all applicable templates. Returns a map of relative output path -> rendered content.

   The template data context must include:
   - `name` (kebab-case), `pascalName`, `camelName`, `label`, `labelPlural`
   - `fields` (array of field configs with name, type, required, etc.)
   - `fieldsArray` (for iteration), `enumFields`, `requiredFields`, `nonStatusFields`
   - `behaviors` (assignable, orderable, etc.)
   - `hasStatusField`, `statusField`, `statusValues`, `hasTransitions`
   - `hasTimestamps`, `hasCreatedAt`, `hasUpdatedAt`
   - `i18nNamespace` (kebab-case feature name)
</action>
