# Plan 999.4-03: CalmDo Migration to Generated/Custom Directory Split

## Objective

Migrate all 5 CalmDo features (tasks, projects, subtasks, work-logs, activity-logs) from the legacy `src/features/` + `convex/` layout to the generated/custom split: `src/generated/{name}/` for generated code, `convex/generated/{name}/` for generated backend, and `src/features/{name}/` retained for custom overrides only. After this plan, the codebase uses the generated/custom split architecture and all 300+ tests pass.

## Why This Matters

The generated/custom split was designed in Phase 999.1 and the pipeline already targets `src/generated/` + `convex/generated/` in "generated" mode. But no real features have ever been migrated to this layout. CalmDo is the proof — if the split works for 5 real entities with complex relationships, behaviors, and 300+ tests, it works for anything.

## must_haves

- [ ] All 5 CalmDo features have generated code in `src/generated/{name}/` and `convex/generated/{name}/`
- [ ] Custom code (hand-written overrides) remains in `src/features/{name}/`
- [ ] `convex/schema.ts` imports from `convex/generated/{name}/` paths
- [ ] All route files import from correct paths (generated or features)
- [ ] All 300+ tests pass after migration
- [ ] `npm test` passes at 100% coverage
- [ ] `npm run typecheck` passes on all 3 tsconfigs

<task id="999.4-03-02">
<title>Diff generated output against existing hand-written code and identify custom overrides</title>
<type>code</type>
<read_first>
- convex/generated/tasks/mutations.ts (just generated)
- convex/tasks/mutations.ts (existing hand-written — has custom logic like status transitions, assignment rules)
- convex/generated/tasks/queries.ts (just generated)
- convex/tasks/queries.ts (existing hand-written — has custom filters, views)
- src/generated/tasks/components/ (just generated)
- src/features/tasks/components/ (existing hand-written — has custom UI logic)
</read_first>
<action>
1. For each of the 5 features, diff the generated output against the existing hand-written code:
   - `convex/generated/{name}/mutations.ts` vs `convex/{name}/mutations.ts`
   - `convex/generated/{name}/queries.ts` vs `convex/{name}/queries.ts`
   - `src/generated/{name}/components/` vs `src/features/{name}/components/`

2. Identify code that exists in hand-written files but NOT in generated output. This is custom code that needs to be preserved in `src/features/{name}/` (the custom override layer).

   Expected custom code (based on CONTEXT.md D-05, D-06):
   - **tasks**: status transition guards, assignment visibility auto-flip, projectId handling, subtask promotion logic, work log time aggregation, activity log creation hooks
   - **projects**: cascade delete (delete project's tasks, work logs), task count aggregation, filtered views by status
   - **subtasks**: reorder logic (position field), promote-to-task mutation, completion count derivation
   - **work-logs**: time parsing, total time computation, ownership validation
   - **activity-logs**: auto-logging helper (`logActivity`), event type enums, entity polymorphic references

3. For each custom piece, create a file in `src/features/{name}/` or `convex/{name}/` that contains ONLY the custom override — not the full CRUD. The override imports from generated and extends/replaces specific functions.

   Pattern for custom mutations:
   ```typescript
   // convex/tasks/mutations.ts (custom layer)
   // Extends generated mutations with domain-specific logic
   import { createBase, updateBase, removeBase } from "../generated/tasks/mutations";
   // ... custom overrides that wrap or replace base mutations
   ```

   If the custom logic is too intertwined with the generated code to separate cleanly, keep the entire file as custom and note it. The goal is to identify what CAN be generated vs what MUST be custom.

4. Document the split decisions in a temporary file `src/features/MIGRATION-NOTES.md` (deleted after migration completes):
   - For each feature: which files are purely generated, which have custom overrides, which are fully custom
</action>
<acceptance_criteria>
- Diff analysis completed for all 5 features
- `src/features/MIGRATION-NOTES.md` exists documenting the split for each feature
- Custom override files identified (may not all be created yet — this is the analysis task)
- Each feature has a clear classification: fully generated files vs custom override files vs fully custom files
- No existing tests are broken by this analysis step (read-only operation)
</acceptance_criteria>
</task>
