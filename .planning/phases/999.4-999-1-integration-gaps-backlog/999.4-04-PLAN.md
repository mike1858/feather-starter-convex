# Plan 999.4-04: feather update CLI Thorough Testing + Plop Unification

## Objective

Verify `feather update` works end-to-end against real projects (after the CalmDo migration), write comprehensive integration tests for all update scenarios, and wire Plop generators to call through to the pipeline so there's a single code generation path. After this plan, both `npx feather generate` and `npm run gen:feature` use the same pipeline, and `feather update` is battle-tested.

## Why This Matters

`feather update` exists but was never tested against real projects — it was built against stubs. Now that Plan 01 replaced stubs with real templates and Plan 03 migrated CalmDo, `feather update` operates on real code. This plan validates the full lifecycle: generate → modify custom code → update → verify custom code untouched. It also unifies the two generation systems (Plop interactive wizard + pipeline programmatic API) so there's one path forward.

## must_haves

- [ ] `feather update` tested end-to-end: regenerates all 5 CalmDo features without breaking tests
- [ ] Uncommitted change detection works correctly
- [ ] `--dry-run` shows accurate preview of what would change
- [ ] `--force` bypasses uncommitted check and regenerates
- [ ] Plop's `gen:feature` calls through to the pipeline (not its own independent wiring)
- [ ] `feather init` has unit tests
- [ ] All update scenarios covered by integration tests
- [ ] `npm test` passes at 100% coverage

<task id="999.4-04-03">
<title>Fix update.ts bug — delete should happen with force flag too</title>
<type>code</type>
<read_first>
- bin/commands/update.ts (updateAction — read the delete logic carefully)
</read_first>
<action>
Fix a bug in `bin/commands/update.ts`:

Current code at line 88: `if (!options.dryRun && !options.force)` guards the delete step. This means when `--force` is used, the old generated directories are NOT deleted before regeneration. This is wrong — `--force` should ALSO delete before regenerating (it's the uncommitted check that force bypasses, not the delete).

Change line 88 from:
```typescript
if (!options.dryRun && !options.force) {
```
To:
```typescript
if (!options.dryRun) {
```

This ensures:
- `--dry-run`: no delete, no write (preview only)
- No flags: delete + regenerate (only if no uncommitted changes)
- `--force`: delete + regenerate (even with uncommitted changes)

Also verify that the `generateFeature` call uses `outputMode: "generated"` (it currently does on line 109).
</action>
<acceptance_criteria>
- `bin/commands/update.ts` line 88 condition is `if (!options.dryRun)` (not `!options.force`)
- Test from Task 01 "update --force overwrites uncommitted changes" passes (force deletes and regenerates)
- `npm test` passes
</acceptance_criteria>
</task>
