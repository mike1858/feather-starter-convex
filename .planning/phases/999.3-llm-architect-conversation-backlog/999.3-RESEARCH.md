## 1. Claude Code Skill Architecture

### How Claude Code Skills Work

Claude Code skills are markdown files in `$HOME/.claude/skills/` (global) or `.claude/skills/` (project-specific). They are loaded automatically when the user invokes the skill via `/skillname` or when Claude detects relevance.

A skill file is a markdown document with:
- **Title/trigger** — when to activate (slash command or contextual)
- **Instructions** — what the skill does, step by step
- **Tool usage** — which tools the skill uses (Bash, Read, Write, Edit, etc.)

Key constraints:
- Skills run in the same Claude Code session — they have access to all tools (Bash, Read, Write, Edit, Grep, Glob)
- Skills do NOT have their own LLM — they ARE the LLM instructions. The developer's AI tool IS the conversation engine.
- No API keys, no SDK imports, no streaming — the skill IS the conversation
- State persistence must use files (JSON checkpoints) since Claude Code sessions may reset

### Skill File Structure

```
.claude/skills/feather-architect.md
```

Contains:
1. Trigger: `/feather:architect` slash command
2. Phase state machine: which conversation phase we're in
3. Conversation prompts for each phase
4. YAML generation instructions
5. Validation integration (calls `feather validate`)

### Implications for This Phase

- **D-04 is confirmed**: No `@anthropic-ai/sdk` dependency. The skill IS the LLM integration.
- The ROADMAP mentions `@anthropic-ai/sdk` for streaming — this is OBSOLETE per D-04/D-05 in CONTEXT.md. The skill approach replaces it entirely.
- Conversation "phases" are NOT network requests — they're sections of the skill's instructions that guide Claude through a structured dialogue.

## 2. Conversation State Management

### JSON Checkpoint Design

Per D-07, conversation state lives in `.feather/conversations/{id}.json`. The checkpoint must capture:

```typescript
interface ConversationState {
  id: string;                    // unique conversation ID
  createdAt: string;             // ISO timestamp
  updatedAt: string;             // ISO timestamp
  currentPhase: 0 | 1 | 2 | 3;  // which phase we're in

  // Phase 0: Entity Discovery
  entities: {
    name: string;
    label: string;
    description: string;
    fields: Record<string, FieldDef>;
    relationships: RelationshipDef[];
  }[];

  // Phase 1: Schema + CRUD
  yamlFiles: Record<string, string>;  // entityName -> YAML content
  validationResults: ValidationResult[];

  // Phase 2: Behavior Overlay
  behaviors: Record<string, BehaviorOverlay>;  // per entity

  // Phase 3: Validation
  sampleData: Record<string, Record<string, unknown>[]>;
  dryRunResult: { success: boolean; errors: string[] } | null;
}
```

### Resume Flow

When resuming a conversation:
1. Skill reads checkpoint file
2. Determines current phase from `currentPhase`
3. Presents summary of what was captured so far
4. Continues from that phase

Per D-08, users can revisit prior phases. This means the checkpoint must support rollback — setting `currentPhase` to an earlier value and optionally clearing downstream data.

### File Naming

Conversation ID should be human-readable: `{project-name}-{timestamp}` or `{domain-slug}-{date}`. Example: `calmdo-2026-03-29.json`.

## 3. The 4-Phase Conversation Protocol

### Phase 0: Entity Discovery

**Input:** Developer's natural language description of their domain
**Method:** WHO/WHAT/WHERE/WHEN/HOW framework from user's Gemini notes
**Output:** Entity list with preliminary relationships

The skill guides the developer through:
1. "Walk me through what your users do, step by step" — identifies actors (WHO) and things they interact with (WHAT)
2. For each step: What info do they need? What do they produce? — identifies entities and fields
3. "Where does this happen? When?" — identifies location/time-dependent entities
4. "How does the process work?" — identifies workflows and status flows

**Smart defaults (Tableau Standard):**
- If user mentions "users" and "items" — auto-suggest a relationship
- If user mentions "status" — auto-suggest an enum field with transitions
- If user mentions "assigned to" — auto-suggest assignable behavior
- If user mentions "created by" — auto-suggest audit trail

### Phase 1: Schema + CRUD

**Input:** Entity list from Phase 0
**Output:** feather.yaml files for each entity, validated by Zod

For each entity:
1. Define fields (type, required, max, default, etc.)
2. Define relationships (belongs_to, has_many, etc.)
3. Define identity (auto-increment, UUID, expression-based)
4. Apply smart defaults from `templates/defaults.yaml`
5. Write feather.yaml to `src/features/{name}/feather.yaml`
6. Validate with `feather validate` (D-10)

**Reference YAML files:** 8 existing feather.yaml files serve as examples:
- `tasks/feather.yaml` — most complex (relationships, behaviors, views, derived data, hooks)
- `todos/feather.yaml` — simplest (basic CRUD)
- `projects/feather.yaml`, `subtasks/feather.yaml`, etc. — various complexity levels

### Phase 2: Behavior Overlay (8 Dimensions)

**Input:** Validated feather.yaml files from Phase 1
**Output:** Enhanced feather.yaml with behavioral specs

Adaptive probing order (from architecture doc section 4.4):
1. **Always ask:** Access Control, Side Effects
2. **Conditional:** Status Flow (if entity has lifecycle), Schedules (if time-dependent)
3. **Always probe:** Cascades (users won't volunteer this)
4. **Then explore:** Derived Data, Custom Views, Integrations

Each dimension maps to a feather.yaml section:
| Dimension | YAML Section | Example |
|-----------|-------------|---------|
| Access Control | `access` | `scope: owner`, `permissions: { create: authenticated }` |
| Status Flow | `statusFlow` or `fields.status.transitions` | State machine |
| Side Effects | `hooks` | `afterSave: custom/tasks/hooks/afterSave` |
| Derived Data | `derivedData` | `subtaskCount: { type: count, source: subtasks }` |
| Custom Views | `views` + `detailView` | View configuration |
| Schedules | `schedules` | Cron expressions, deadlines |
| Integrations | `integrations` | Webhook, email triggers |
| Identity & Naming | `identity` | Auto-increment, expression-based |

### Phase 3: Validation

**Input:** Complete feather.yaml files
**Output:** Validated, generation-ready YAML

Three validation layers (D-10, D-11, D-12):
1. **Zod validation** — `feather validate` on all generated YAML files
2. **Sample data test** — generate 2-3 example rows per entity, display as table
3. **Dry-run generation** — `feather generate --dry-run` to verify generator compatibility

Also: "What reports do you need?" question to validate data model completeness.

## 4. Web Preview Dashboard

### Requirements (D-01, D-02, D-03)

A web preview opens in the browser showing:
- **ER diagram** — entity relationships (mermaid.js is the natural choice — already widely used, no extra deps)
- **Generated YAML** — syntax-highlighted display of current feather.yaml content
- **Sample data tables** — HTML tables with generated sample rows
- **Validation status** — pass/fail for each entity's YAML

Dashboard updates **between conversation phases** (not live streaming). User can also trigger **on-demand refresh** mid-conversation.

### Implementation Options

| Option | Mechanism | Pros | Cons |
|--------|-----------|------|------|
| Static HTML + file watcher | Generate HTML, `open` in browser, watch for changes | Simple, no server | Manual refresh (or basic polling) |
| Lightweight dev server | Express/Hono on a port, serve dashboard | Auto-refresh via SSE/WebSocket | Extra dependency, port management |
| Vite plugin | Hook into existing Vite dev server | Hot module replacement | Couples to project's dev setup |

**Recommendation:** Static HTML with auto-refresh polling. Simplest approach, no dependency on project's dev setup, works with any AI tool. The skill writes HTML to `.feather/preview/index.html` and opens it in the browser. A small `<script>` in the HTML polls for file changes.

### Mermaid.js for ER Diagrams

Mermaid.js can render entity-relationship diagrams directly in HTML:
```html
<script src="https://cdn.jsdelivr.net/npm/mermaid/dist/mermaid.min.js"></script>
<div class="mermaid">
erDiagram
    TASKS ||--o{ SUBTASKS : has
    TASKS ||--o{ WORK_LOGS : has
    PROJECTS ||--o{ TASKS : contains
</div>
```

No npm dependency needed — CDN link in the HTML file.

### YAML Syntax Highlighting

Prism.js or highlight.js via CDN for YAML syntax highlighting in the preview.

| Existing feather.yaml files | `src/features/*/feather.yaml` | Reference examples shown to user |

### Validation Auto-Fix (D-13)

When Zod validation fails, the skill should:
1. Parse the error messages
2. Auto-fix the YAML (e.g., add missing required fields, fix types)
3. Show the user a diff of what changed
4. Ask to confirm

This is straightforward since the skill is the LLM — it can read the error, fix the YAML, and explain the fix in one turn.

## 6. Cascade Behavior (D-08, D-09)

When a user revisits an earlier phase (e.g., adds an entity during Phase 2):

1. Detect what changed (new entity, modified fields, etc.)
2. Identify downstream impacts (relationships to update, views to adjust, etc.)
3. **Infer** cascade updates based on prior patterns (Tableau Standard)
4. Present cascaded changes to user for confirmation
5. Apply confirmed changes

Example: User adds a "categories" entity in Phase 0 while in Phase 2.
- Skill auto-suggests: add `categoryId` field to tasks, add `belongs_to` relationship
- Skill shows the proposed YAML changes
- User confirms or modifies

## 8. Validation Architecture

### Verification Approach

| What to Verify | How | Automated? |
|---------------|-----|-----------|
| Skill loads and responds to `/feather:architect` | Manual — invoke in Claude Code | No |
| Phase 0 produces entity list | Run against CalmDo description | Semi (can script domain input) |
| Phase 1 produces valid YAML | `feather validate` on output | Yes |
| Phase 2 adds behavioral specs | Check YAML has access/hooks/etc sections | Yes (grep) |
| Phase 3 dry-run succeeds | `feather generate --dry-run` | Yes |
| Checkpoint saves/resumes | Write state, restart, verify resume | Semi |
| Web preview renders | Open HTML, check for mermaid diagram | Manual (agent-browser) |
| Cascade on phase revisit | Add entity in Phase 2, verify downstream updates | Manual |

### Unit-Testable Components

- Conversation state serialization/deserialization
- YAML generation from entity definitions
- Mermaid diagram generation from relationships
- Sample data generation
- Dashboard HTML generation

### Integration Test: CalmDo

The CalmDo domain (5 entities: tasks, projects, subtasks, work-logs, activity-logs) is the integration test case. The conversation should produce 5 feather.yaml files that match or exceed the existing ones.
