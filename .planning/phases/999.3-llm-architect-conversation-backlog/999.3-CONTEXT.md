<decisions>
## Implementation Decisions

### Conversation Runtime
- **D-01:** CLI + web preview hybrid. Conversation runs in terminal via Claude Code skill (`/feather:architect`). A web preview opens in the browser showing a full dashboard.
- **D-02:** Web preview dashboard shows: ER diagram (entity relationships), generated YAML with syntax highlighting, sample data tables, and validation status.
- **D-03:** Dashboard updates between conversation phases (not live streaming). User can also trigger an on-demand refresh mid-conversation.

### LLM Integration
- **D-04:** Claude Code skill, NOT a standalone API client. No `@anthropic-ai/sdk` dependency needed. The LLM is provided by whatever AI tool the developer uses (Claude Code, Cursor, etc.).
- **D-05:** No API key management needed. The developer's existing AI tool handles authentication.
- **D-06:** Claude Code only for v1. Portability to other tools (Cursor, Windsurf, Aider) deferred until demand exists.

### Conversation State
- **D-07:** JSON checkpoint files in `.feather/conversations/{id}.json`. After each phase, conversation state is persisted (entities discovered, fields defined, behaviors configured, etc.). On resume, skill reads checkpoint and continues from last completed phase.
- **D-08:** User can revisit any prior phase (e.g., add an entity during Behavior Overlay). Non-linear navigation supported.
- **D-09:** When revisiting an earlier phase, downstream phases auto-cascade with confirmation. Skill infers changes based on prior patterns, user confirms each cascaded update. Follows Tableau Standard: infer first, let user override.

### Output Validation (3-Layer Progressive)
- **D-10:** Layer 1 — Zod schema validation after every phase. Uses existing `feather validate` (bin/commands/validate.ts). Catches structural errors immediately.
- **D-11:** Layer 2 — Sample data test during Phase 3 (Validation). Generate 2-3 example rows per entity, display as table, user confirms the model captures the right data.
- **D-12:** Layer 3 — Dry-run generation at the end. Run `feather generate --dry-run` to prove the YAML produces valid code. Catches generator-level issues.
- **D-13:** On Zod validation failure: auto-fix the YAML error, show the user a diff of what changed, ask to confirm. Tableau Standard — fix it, don't ask what to do.

### Claude's Discretion
- Web preview implementation (standalone dev server vs Vite plugin vs static HTML with file watcher)
- ER diagram rendering library (mermaid.js, d3, or custom SVG)
- Conversation checkpoint JSON schema (internal detail)
- Phase transition UX (how the skill signals phase completion and prompts for next phase)
- Claude Code skill file structure and hook integration

</decisions>

## Existing Code Insights

### Reusable Assets
- **`bin/commands/validate.ts`**: Zod-based YAML validation — directly reusable for D-10 (Layer 1 validation after each phase)
- **`bin/commands/generate.ts`**: Generator pipeline — reusable for D-12 (dry-run generation)
- **8 feather.yaml files**: Real-world examples the skill can reference to show users what "good" YAML looks like
- **`templates/defaults.yaml`**: Smart defaults for all YAML sections — skill should apply these automatically (Tableau Standard)

### Established Patterns
- **Commander.js CLI**: `bin/feather.ts` uses Commander.js — but the architect is a Claude Code skill, not a CLI command. Different packaging.
- **Feature-folder architecture**: Each entity = one folder + one feather.yaml. The conversation naturally maps to "one entity at a time."
- **Zod as single source of truth**: YAML schema validated by Zod, same validators used for runtime validation during generation.

### Integration Points
- **Claude Code skill directory**: Skills live in `$HOME/.claude/skills/` or project `.claude/skills/`
- **`feather validate`**: Skill calls this after each phase for structural validation
- **`feather generate --dry-run`**: Skill calls this at end for code generation validation
- **`.feather/conversations/`**: New directory for conversation checkpoint JSON files

</code_context>

<specifics>
## Specific Ideas

- The key insight is that users already have an AI tool running — the architect should leverage that tool's LLM rather than managing its own API connection. This eliminates API key management entirely.
- Conversation follows user's original WHO/WHAT/WHERE/WHEN/HOW framework for entity discovery (Phase 0), then maps to the 8-dimension behavior overlay
- Web preview dashboard is the "show, don't tell" component — users see their domain model forming in real time (between phases)
- CalmDo (5 entities with relationships) should be the integration test case for the conversation flow

</specifics>

<deferred>
## Deferred Ideas

- **Multi-tool portability** (Cursor, Windsurf, Aider) — Claude Code only for v1. Port later if demand.
- **Standalone API mode** (`@anthropic-ai/sdk`) — originally in roadmap, replaced by skill approach. Could be added as fallback for non-Claude-Code users.
- **Live streaming dashboard updates** — v1 updates between phases only. Live updates would need websocket/SSE.
- **Collaborative architect sessions** — multiple developers designing the same domain together.

</deferred>
