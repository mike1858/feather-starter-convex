# Plan 999.3-02: Claude Code Skill — /feather:architect Conversation Engine

## Objective

Create the Claude Code skill file that implements the 4-phase LLM architect conversation. The skill guides a developer through describing their business domain in plain English and produces validated `feather.yaml` files. This is the core deliverable of Phase 999.3 — the skill IS the conversation engine (D-04: no API SDK needed, the developer's AI tool provides the LLM).

## Why This Matters

This skill is the primary interface for domain modeling in Feather. Instead of hand-writing YAML, developers have a guided conversation that produces correct output. The skill leverages D-04/D-05 (no API key management), D-07 (checkpoint persistence), D-08 (non-linear phase navigation), D-10/D-11/D-12 (3-layer validation), and the WHO/WHAT/WHERE/WHEN/HOW entity discovery framework.

## must_haves

- [ ] `.claude/skills/feather-architect.md` skill file invokable via `/feather:architect`
- [ ] Phase 0: Entity Discovery using WHO/WHAT/WHERE/WHEN/HOW framework
- [ ] Phase 1: Schema + CRUD generation producing valid feather.yaml per entity
- [ ] Phase 2: Behavior Overlay with adaptive probing across 8 dimensions
- [ ] Phase 3: Validation with sample data, Zod validation, and dry-run generation
- [ ] Conversation state checkpoint after each phase (uses Plan 01 checkpoint system)
- [ ] Resume support: skill detects existing conversation and offers to continue
- [ ] Phase navigation: user can revisit prior phases with cascade confirmation (D-08, D-09)
- [ ] Smart defaults from `templates/defaults.yaml` applied automatically (Tableau Standard)
- [ ] YAML generation helper that produces schema-valid feather.yaml content
- [ ] Sample data generation helper for Phase 3 validation
- [ ] `npm test` passes at 100% coverage for helper modules

   - Produces a complete feather.yaml string in YAML format

**Skill file structure:**

```markdown
# /feather:architect — Domain Modeling Conversation

## Trigger
When the user types `/feather:architect` or asks to "design a feature", "model my domain", "create entities from description".

## Overview
Guide the developer through a 4-phase conversation to produce validated feather.yaml files from a plain English domain description.

## On Start
1. Check for existing conversations: run `ls .feather/conversations/*.json 2>/dev/null`
2. If conversations exist, offer: "Resume {id} (Phase {N})?" or "Start new conversation"
3. If starting new: ask for domain name, create conversation with `createConversation(domain)`
4. If resuming: load checkpoint, display summary, continue from current phase

## Phase 0: Entity Discovery
[WHO/WHAT/WHERE/WHEN/HOW framework instructions]

Ask the developer:
"Walk me through what your users do, step by step."

For each step mentioned:
- WHO: Who performs this action? (actors → entities)
- WHAT: What information do they need or produce? (data → fields)
- WHERE: Where does this happen? (locations → entities or fields)
- WHEN: When does this happen? (time → date fields, schedules)
- HOW: How does the process work? (workflows → status flows)

Build entity list progressively. After each entity is discovered:
- Name it (kebab-case)
- Label it (human readable)
- List preliminary fields
- Identify relationships to other entities

When the developer says they're done describing their domain:
- Display the entity list as a summary table
- Ask: "Does this capture your domain? Anything missing?"
- If confirmed: save checkpoint, advance to Phase 1

## Phase 1: Schema + CRUD
[For each entity, define complete schema]

For each entity from Phase 0:
1. Review fields — add types, required flags, defaults, max lengths
2. Apply smart defaults from templates/defaults.yaml
3. Define relationships formally (belongs_to, has_many with column names)
4. Set identity strategy (auto-increment by default)
5. Generate feather.yaml using `generateFeatherYaml()`
6. Validate with `npx tsx bin/feather.ts validate <file>`
7. If validation fails: auto-fix the YAML (D-13), show diff, confirm

After all entities are defined:
- Show complete entity-relationship summary
- Ask: "Ready to add behaviors and business rules?"
- Save checkpoint, advance to Phase 2

## Phase 2: Behavior Overlay (8 Dimensions)
[Adaptive probing — ask relevant questions per entity]

For each entity, probe in this order:
1. **Access Control** (always ask): "Who can see and modify {entity}?"
   → scope, permissions, sharing
2. **Side Effects** (always ask): "When a {entity} changes, what else should happen?"
   → hooks, cascades, notifications
3. **Status Flow** (if entity has enum fields): "Does {entity} have a lifecycle?"
   → transitions, guards
4. **Schedules** (if time-dependent indicators): "Any time-based automation for {entity}?"
   → cron, deadlines
5. **Cascades** (always probe — users won't volunteer): "If you delete a {parent}, what happens to its {children}?"
   → cascade rules in hooks
6. **Derived Data**: "Any calculated fields? Counts, sums, averages from related data?"
   → derivedData section
7. **Custom Views**: "How should {entity} be displayed? List? Cards? Kanban?"
   → views section
8. **Integrations** (if business indicates external systems): "Any external systems involved?"
   → integrations section

After each entity's overlay:
- Update the feather.yaml with new sections
- Re-validate
- Show what was added

Save checkpoint after all entities overlaid, advance to Phase 3

## Phase 3: Validation
[Three-layer validation per D-10/D-11/D-12]

1. **Zod validation** (D-10):
   Run `npx tsx bin/feather.ts validate src/features/` on all generated YAML
   Show results. Auto-fix any failures (D-13).

2. **Sample data test** (D-11):
   Generate 2-3 example rows per entity using `generateSampleRows()`
   Display as markdown tables
   Ask: "Does this data look right? Does it capture what your {entity} should hold?"

3. **Dry-run generation** (D-12):
   Run `npx tsx bin/feather.ts generate {entity} --dry-run` for each entity
   Show files that would be generated
   Report any generation errors

4. **Reports question**:
   "What reports or aggregations do you need? This validates your data model captures the right relationships."

After validation passes:
- Display final summary: entities, fields, relationships, behaviors
- Save final checkpoint
- Offer: "Ready to generate? Run: feather generate {entity} for each entity"

## Phase Navigation (D-08, D-09)
At any point, the user can say "go back to Phase {N}" or "add another entity".
- Save current state
- Set currentPhase backward
- If clearDownstream requested: show what will be cleared, confirm
- If not: preserve downstream data, just reopen the phase for additions

## Cascade Detection (D-09)
When revisiting Phase 0 (adding/modifying entities):
- Detect what changed
- For each change, infer downstream impacts:
  - New entity: suggest relationships to existing entities
  - Modified fields: update YAML, re-validate
  - Removed entity: warn about broken relationships
- Show proposed cascades, confirm before applying
