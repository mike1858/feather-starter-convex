<code_context>
## Existing Code Insights

### Reusable Assets
- **`e2e/fixtures.ts` + `e2e/helpers.ts`**: Existing Playwright helpers — extend rather than replace.
- **Dev mailbox**: `convex/devEmails/` captures all OTP and password reset emails in dev mode. E2E tests can query this table to get codes programmatically.
- **`renderWithRouter` + `ConvexTestClient`**: Frontend test helpers already exist for unit/integration tests. E2E tests complement these with real browser interaction.

### Established Patterns
- **Playwright webServer config**: `npm run test:e2e` auto-starts dev server (already configured).
- **100% coverage enforcement**: Pre-commit hook runs `vitest run --coverage`. E2E tests are separate (Playwright, not Vitest).
- **Feature-folder test co-location**: Unit/integration tests live next to source. E2E tests live in `e2e/` (separate, as they test full flows).

### Integration Points
- **`npm run dev`**: Starts both Vite frontend and Convex backend — E2E tests need both running.
- **`npx convex dev`**: Backend dev server — needed for real data operations in E2E.
- **Dev mailbox API**: Public mutations for devEmails — E2E can read captured OTP codes.

</code_context>

## Specific Ideas

- Dev mailbox as the E2E auth helper is a clean pattern — no need to mock email or bypass auth. The infrastructure already exists.
- E2E tests validate what unit tests can't: real browser rendering, navigation flows, form submission, Convex reactivity.
- This is the final quality gate before the team uses the product — every user-facing flow must work end-to-end.
