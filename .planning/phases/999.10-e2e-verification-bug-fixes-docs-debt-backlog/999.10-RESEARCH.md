### 2. TS2589 Type Depth Bug

**Current state:** Both `convex/otp/ResendOTP.ts` (line 14) and `convex/password/ResendOTPPasswordReset.ts` (line 22) use `// @ts-ignore TS2589` to suppress the type error.

**Root cause:** `ConvexHttpClient.mutation()` with deep `api.devEmails.mutations.store` type produces excessive type instantiation depth. This is a known Convex/TypeScript interop issue where the generated API types are deeply nested.

**TypeScript check:** `tsc --noEmit -p convex/tsconfig.json` passes cleanly (the `@ts-ignore` suppresses it).

**Assessment:** This is a legitimate suppression — the type depth error is in the Convex generated types, not in user code. The `@ts-ignore` with descriptive comment is the standard workaround. Fixing this "properly" would require either:
- Casting `client.mutation` to `any` (worse)
- Creating a typed wrapper (over-engineering for 2 call sites)
- Waiting for Convex to fix their type depth in generated API types

**Recommendation:** Keep the `@ts-ignore` with clear comment. This is not a real bug — it's a TypeScript limitation with deeply nested generated types. Document it as an accepted suppression.

### 3. Barrel Export Gaps

**Current state:**
- `src/features/subtasks/index.ts` — contains only a comment: `// Subtasks are accessed through the TaskDetailPanel, not as a standalone page.`
- `src/features/work-logs/index.ts` — contains only a comment: `// Work logs are accessed through the TaskDetailPanel, not as a standalone page.`
- `src/features/activity-logs/index.ts` — **does not exist at all** (directory only has `feather.yaml`)

**Context:** These features don't have standalone frontend pages — their components live inside `src/features/tasks/components/` (SubtaskList.tsx, SubtaskItem.tsx, WorkLogForm.tsx, WorkLogList.tsx). Activity logs don't have separate frontend components either — the timeline is embedded in the task detail panel.

**Assessment:** These are not "gaps" that need fixing — they're by design. Subtasks, work logs, and activity logs are child entities of tasks, rendered within the TaskDetailPanel. There's nothing meaningful to export from their barrel files. The placeholder comments are accurate.

**Options:**
1. Delete the placeholder index.ts files entirely (they're misleading — they suggest something should be there)
2. Keep them with the comment (current state, OK)
3. Export the components that live in tasks/components/ from these barrels (creates confusing cross-feature imports)

**Recommendation:** Option 1 or 2. These features don't need barrel exports because they're not independently consumed.
