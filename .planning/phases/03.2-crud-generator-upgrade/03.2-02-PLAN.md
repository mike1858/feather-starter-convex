    - `reorder` mutation (if `orderable: true`): auth guard, update position field
    - Wrap each mutation in `// @generated-start {mutationName}` markers
    - Include `// @custom-start {mutationName}-pre` and `// @custom-start {mutationName}-post` hooks inside each handler

    - For each `filteredViews` entry: generate a named query with preset filters applied
    - Wrap in `// @generated-start` / `// @generated-end` markers

    **Template 1: `templates/feature/mutations.test.ts.hbs`**

    Produce a file matching `convex/tasks/mutations.test.ts` structure:

    ```handlebars
    // @generated-start imports
    import { describe, expect } from "vitest";
    import { api } from "../_generated/api";
    import { test } from "../test.setup";
    // @generated-end imports

    // @custom-start imports
    // @custom-end imports
    ```

    Test blocks (inside `describe` per mutation):

    1. **create** describe block:
       - `"creates a {label} with default values"` — call create mutation with minimal args (just required fields), verify record exists in DB via `testClient.run(async (ctx: any) => ctx.db.query("{name}").collect())`, assert required fields match, assert defaults applied (status default, timestamps, userId)
       - `"does nothing when unauthenticated"` — call via `testClient.mutation(...)` (no auth), verify no records created
       - For each required field: `"rejects when {field} is missing"` — call with missing required field, expect error or no record

    2. **update** describe block:
       - `"updates {field} fields"` — create a record, call update with changed values, verify fields updated via DB query
       - `"does nothing when unauthenticated"` — call update via testClient, verify no change

    3. **remove** describe block:
       - `"deletes a {label}"` — create, remove, verify gone from DB
       - `"does nothing when unauthenticated"` — verify record still exists after unauthenticated remove
       - If `softDelete: true`: `"soft deletes by setting deletedAt"` — verify record exists but has deletedAt set

    4. **updateStatus** describe block (if transitions defined):
       - For each valid transition: `"transitions from {from} to {to}"` — create with status {from}, call updateStatus, verify new status
       - `"rejects invalid transition"` — try an invalid transition, expect error

    5. **assign** describe block (if assignable):
       - `"assigns to another user"` — create, assign to different userId, verify assigneeId changed

    6. **reorder** describe block (if orderable):
       - `"updates position"` — create, reorder with new position, verify position changed

    Wrap each describe in `// @generated-start test-{mutation}` markers.
