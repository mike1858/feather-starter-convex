# Phase 03.2: CRUD Generator Upgrade - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Upgrade Plop.js generators from TODO stubs to a comprehensive, Frappe-level CRUD generation system. The generator output should be a fully working, demoable feature covering 80% of typical client needs — the agent only customizes the genuinely domain-specific 20%. Generators support a YAML-first workflow where features are defined declaratively and can be re-run as the data model evolves during client sessions.

</domain>

<decisions>
## Implementation Decisions

### Generator Landscape

- **D-01:** Unified `gen:feature` produces everything: Zod schema, backend CRUD (mutations + queries + tests), frontend (multiple view types, form, detail page, item component, tests), route file, i18n locale files, and auto-wiring of shared files.
- **D-02:** Phoenix-style decomposition — each layer also available standalone:
  - `gen:feature` — all-in-one (like `phx.gen.live`)
  - `gen:schema` — Zod schema only (new)
  - `gen:backend` — Convex mutations + queries + tests (new)
  - `gen:frontend` — Page components, item, list (new)
  - `gen:form` — Form component wired to Zod schema (upgraded, merged into gen:feature)
  - `gen:route` — Route file (existing, keep as-is)
  - `gen:convex-function` — Single Convex function (existing, keep as-is)
- **D-03:** `gen:form` removed as standalone — merged into `gen:feature`. All other standalone generators kept.

### YAML-First Workflow

- **D-04:** Features defined via YAML definition files. Two entry points:
  - **Interactive wizard** — prompts for name + fields one-by-one with add/edit/remove before generating. Writes the YAML file.
  - **Direct YAML** — write/edit YAML by hand, run generator from it.
- **D-05:** YAML files co-located with features: `src/features/{name}/{name}.gen.yaml`
- **D-06:** Three-file chain for transparency:
  - `templates/defaults.yaml` — shared defaults for all generators (single source of truth)
  - `src/features/{name}/{name}.gen.yaml` — lean overrides (only what differs from defaults)
  - `src/features/{name}/{name}.resolved.yaml` — merged output (generated, committed to repo)
- **D-07:** Smart merge on re-run — generator can be re-run repeatedly as data model evolves (client session workflow). Needs research into implementation approach (marked regions, AST-based merge, etc.).

### Field Definition

- **D-08:** Rails-style field prompting with types: `name:string, status:enum(active,on_hold), description:text?`
- **D-09:** Core field types: `string` (default), `text`, `boolean`, `number`, `enum(val1,val2,...)`
- **D-10:** Optional modifier via `?` suffix: `description:text?` → `z.string().optional()`
- **D-11:** Convention-over-configuration with smart defaults:
  - Field type defaults to `string`
  - Fields default to optional (`required: false`)
  - String max defaults to 200, text max defaults to 5000
  - Boolean defaults to `false`
  - All CRUD operations generated by default
  - Timestamps (createdAt + updatedAt) auto-included
  - Auth guard auto-included
  - All fields shown on all views by default
- **D-12:** Bare field syntax — a field name with no properties inherits all defaults: `just_title:` is a valid field (string, optional, max 200).
- **D-13:** Enum fields support color mapping — auto-assigned by generator (overridable in YAML).
- **D-14:** Enum fields support status transitions — optional `transitions:` map defines valid state changes. Generator produces `updateStatus` mutation with validation.

### Timestamps

- **D-15:** Timestamps support three modes: `both` (default), `createdAt` only, `updatedAt` only, or `false`.

### Field Visibility

- **D-16:** Fields support both `showOn` and `hideOn` for view-specific visibility: `hideOn: [list]` hides from list page. If neither specified, field shown everywhere.

### Relationships

- **D-17:** Two relationship types following Frappe's model:
  - `children` (tight composition) — child CANNOT exist without parent. Saved atomically, deleted together, no independent routes. Child fields defined inline in parent YAML. Renders as inline editable grid on parent detail page. (PO line items, invoice details)
  - `has_many` (loose association) — child exists independently with optional/required link to parent. Child has own routes, forms, list page. `required: true/false` controls whether child must belong to parent. `onDelete: cascade|nullify|restrict`.
- **D-18:** `belongs_to` — reverse of has_many. Auto-creates foreign key field + index. Supports custom column names (`column: creatorId`) and custom table references (`table: users`).
- **D-19:** `embedded` — nested objects stored in parent document (no separate table). For small, bounded data always read with parent (~10 items max per Convex guidance). Uses `v.object()` in Convex schema.
- **D-20:** Separate tables is the default for parent-child patterns in Convex. Convex mutations are ACID transactions across tables — no atomicity trade-off vs embedding.

### Auto-Indexes

- **D-21:** Indexes auto-generated from conventions: `by_userId` (always), `by_{relation}` for belongs_to, and for fields with `filterable: true` or enum type. Resolved YAML shows computed indexes. Custom indexes can be added in feature YAML.

### Views & Pagination

- **D-22:** Generator produces multiple view types with a view switcher: `list` (text rows, infinite scroll), `card` (visual grid, infinite scroll), `table` (columnar with sorting, offset pagination), `cover` (image background cards). Inspired by Glide Apps.
- **D-23:** Pagination mode tied to view type: table → offset pagination (default 20/page), list/card/cover → infinite scroll.
- **D-24:** Declarative filtered views — YAML can define named views with preset filters, each getting its own route and nav entry (e.g., "My Tasks" with `assigneeId: $currentUser`, "Team Pool" with `assigneeId: null`).

### Behaviors (Toggle-Based Features)

- **D-25:** `softDelete: true` — adds `deletedAt` field, `remove` sets it instead of deleting, queries auto-filter deleted records, "restore" mutation, "show deleted" toggle.
- **D-26:** `auditTrail: true` — creates activity log table, auto-logs CRUD events + status changes, adds activity timeline to detail page.
- **D-27:** `immutable: true` — adds `committedAt` + `commit` mutation, locked after commit, "Draft"/"Committed" badge, read-only form for committed records.
- **D-28:** `assignable: true` — adds `assigneeId` field (optional, references users), user dropdown in form, assignee filter on list page, "My Items" vs "All Items" toggle, `by_assignee` index.
- **D-29:** `orderable: true` — adds `position` field (auto-assigned), drag handles on list items via dnd-kit, reorder mutation, queries sorted by position.
- **D-30:** Three-layer access model in YAML, inspired by Frappe (3 layers), Phoenix 1.8 scopes (query filtering), and Pundit (simple policy methods):
  ```yaml
  access:
    scope: owner          # owner | team | public | custom
    permissions:
      create: authenticated  # authenticated | owner | role(roleName) | public
      read: owner
      update: owner
      delete: owner
    sharing: false         # true adds per-record sharing table
  ```
  - **Scope** (Layer 1): Query filtering — `owner` filters by `userId == currentUser`, `team` shows all to authenticated users, `public` has no auth guard, `custom` leaves logic to agent.
  - **Permissions** (Layer 2): Action authorization — `authenticated` requires login, `owner` requires record ownership, `role(admin)` checks user role (future RBAC), `public` allows anyone.
  - **Sharing** (Layer 3): Per-record grants — when `sharing: true`, generates a shares table, share/unshare mutations, query includes shared records, "Share" button on detail page.
  - Research sources: Frappe (role→DocType matrix, User Permissions, DocShare), Phoenix 1.8 (Scope struct for query filtering), Pundit (policy classes with boolean methods), CanCanCan (auto-SQL from abilities), Spring Security (annotation-based), Shiro (wildcard strings).

### Wiring Automation

- **D-31:** Generator auto-wires all shared files: `convex/schema.ts` (table definition), `src/shared/nav.ts` (nav entry), `src/shared/errors.ts` (error group), `src/i18n.ts` (namespace), `public/locales/{en,es}/{name}.json` (translations).
- **D-32:** Auto-wire with transparency — generator prints a diff summary of every shared file it modified.
- **D-33:** No `gen:remove` command — manual cleanup on feature deletion is acceptable.

### Testing

- **D-34:** Generated tests follow existing project patterns:
  - Backend: `test` from `convex/test.setup.ts`, `client` (authenticated), `testClient` (unauthenticated + direct DB), `describe` blocks per function
  - Frontend: `test` from `@cvx/test.setup`, `renderWithRouter`, `userEvent.setup()`
  - Verification via `testClient.run(async (ctx: any) => ...)` for backend state, `client.query(...)` for frontend
  - Always test both authenticated and unauthenticated paths
  - Behavior-based assertions (no mock call counts)
- **D-35:** Generated tests cover: auth guard, all CRUD operations, field validation, status transitions (if defined), rendering, empty state. Must achieve 100% coverage of generated code.

### Tasks Feature Alignment

- **D-36:** All 6 tasks-vs-generator differences become generator features (not agent customizations):
  - Visibility/permissions → three-layer access model (scope + permissions + sharing)
  - Assignment → behavior (`assignable: true`)
  - Status transitions → YAML field option
  - Drag-to-reorder → behavior (`orderable: true`)
  - Multiple filtered views → YAML views feature
  - Status badge colors → auto-assigned from enum values
- **D-37:** Create retroactive `tasks.gen.yaml` documenting the tasks data model. Refactor tasks structure where it diverges from generator pattern; keep domain-specific customizations as-is.

### Claude's Discretion

- Smart merge implementation approach (marked regions, AST-based, Plop custom actions)
- Component styling details (padding, shadows, hover states) — follow existing UI patterns
- Default color palette for enum badge auto-assignment
- View switcher component design
- Inline editable grid component for `children` relationship type
- How `gen:schema`, `gen:backend`, `gen:frontend` share templates with `gen:feature`

<specifics>
## Specific Ideas

- **Frappe-level comprehensiveness** — Generator output should be demoable to a client immediately. 80% of needs handled by generator, agent customizes the 20%.
- **Client session workflow** — User iterates on data model with clients: edit YAML → re-run generator → client reviews in browser → edit YAML → re-run. Smart merge preserves customizations.
- **`templates/defaults.yaml`** as single reference file — team members check defaults here instead of guessing. Feature YAMLs stay lean, resolved YAMLs show everything.
- **Ash Framework inspiration** — "declare once, derive the rest." Smart defaults where the user only specifies what differs from convention.
- **Glide Apps inspiration** — Multiple view types (list, card, table, cover) with view switcher. Each view type has its own pagination mode and field display.
- **Frappe inspiration** — `children` (tight composition, istable) vs `has_many` (loose association, Link fields). Clear separation of ownership semantics.

</specifics>

<deferred>
## Deferred Ideas

- **Full RBAC system** — The `access.permissions` YAML supports `role(roleName)` but v2.0 has no roles/RBAC infrastructure. Role-based authorization is deferred to v3.0+.
- **Kanban board view** — Glide has it, CalmDo roadmap lists it for future phase. Not in generator v1.
- **Calendar view** — Useful for date-oriented entities. Not in generator v1.
- **Data grid view** (spreadsheet-like inline editing) — Power user feature. Not in generator v1.
- **`feather install <module>` CLI** — Unified installer from review feedback (P1 todo). Separate effort.
- **Config-driven assembler** — v3.0 future requirement (CFG-01 through CFG-04). Build features first, extract patterns second.
- **Nested children (child of child)** — Frappe doesn't support it either. Not in v1.
