## Phase Boundary

</domain>

<decisions>
## Implementation Decisions

### Exclusion Triage Criteria
- **D-01:** Simple rule: **if a file has logic (conditionals, event handlers, state changes), it gets tested.** Pure pass-through wrappers (no props transformation, no conditional rendering) may stay excluded.
- **D-02:** Portal component exclusions (TaskDetailPanel, SubtaskItem, WorkLogForm, etc.) — find a way to test them. Render the component directly outside the portal, use testing-library's `within()` for portal content, or extract logic into a testable hook. "jsdom can't" is usually solvable.
- **D-03:** Legitimate exclusions that stay: auto-generated (`convex/_generated/**`, `routeTree.gen.ts`), test infrastructure (`test.setup.ts`, `test-helpers.tsx`), framework entrypoints (`main.tsx`, `app.tsx`, `router.tsx`), type-only files, barrel re-exports with no logic.

### v8 Ignore Purge Strategy
- **D-04:** Blanket purge — remove ALL 154 `v8 ignore` directives across the codebase. Run coverage. Whatever drops below 100%, write tests for it.
- **D-05:** During the fix pass, dead code (fallbacks that can never be reached) should be DELETED rather than tested. No point testing code that shouldn't exist.

### Generated Example Apps Scope
- **D-06:** Example apps (todos, tickets, contacts) excluded from THIS audit. They're about to be regenerated by Phase 03.2.1.1. After regeneration completes, they get brought into full coverage.
- **D-07:** Same test quality standard for all features — CalmDo and examples are treated identically. The separation between "production" and "example" is a 999.2 concern, not a testing concern.

### Audit Depth
- **D-08:** Full re-audit of ALL test files against the 12-point testing philosophy checklist. Not just exclusion cleanup — verify Phase 7's MECE rewrite held up.
- **D-09:** Two-pass approach: (1) Catalogue — audit every test file, produce findings report with severity (must-fix, should-fix, suggestion). (2) Fix — implement all must-fix and should-fix items.

### Claude's Discretion
- Order of file audits (by feature, by severity, by type)
- How to restructure portal component tests (direct render vs within() vs hook extraction)
- Grouping fixes into commits (by feature vs by fix type)

</decisions>
