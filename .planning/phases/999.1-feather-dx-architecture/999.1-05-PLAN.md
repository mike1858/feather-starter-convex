# Plan 999.1-05: Feather CLI — Command-Line Interface and feather update

## Objective

Build the `feather` CLI that wraps the generator pipeline, adds the `feather update` command for safe upstream regeneration, and provides `feather validate` for YAML verification. This replaces `npm run gen:feature` with a more capable command that reads YAML and orchestrates the full pipeline.

## Why This Matters

The CLI is the developer-facing entry point. It ties together YAML validation, code generation, AST wiring, and safe updates into a cohesive experience. Without it, developers must manually orchestrate Plop + wiring + validation — the exact fragile workflow that causes generator bugs.

## must_haves

- [ ] `feather generate <name>` reads YAML, validates, scaffolds, and wires a feature
- [ ] `feather generate --from-yaml <path>` generates from an existing YAML file
- [ ] `feather update` regenerates all `generated/` files from current YAML configs
- [ ] `feather validate [path]` validates YAML files and reports errors with line numbers
- [ ] `feather init` creates root `feather.yaml` and directory structure
- [ ] CLI uses commander for argument parsing
- [ ] CLI is registered as `feather` bin in package.json
- [ ] Tests cover all commands (generate, update, validate, init)
- [ ] `npm test` passes at 100% coverage

```typescript
export const generateCommand = new Command("generate")
  .description("Generate a feature from YAML specification")
  .argument("[name]", "Feature name (creates new YAML + scaffolds)")
  .option("--from-yaml <path>", "Generate from existing YAML file")
  .option("--legacy", "Output to features/ directory instead of generated/")
  .option("--dry-run", "Preview files without writing")
  .option("--skip-wire", "Skip AST wiring step (scaffold only)")
  .action(async (name, options) => { ... });
```

Two modes:

**Interactive (no --from-yaml):**
1. Prompt for feature name (if not provided as argument)
2. Create minimal YAML file at `src/features/{name}/feather.yaml`
3. Validate YAML
4. Run pipeline (scaffold + wire)
5. Report created files

**From YAML (--from-yaml):**
1. Read specified YAML file
2. Validate
3. Run pipeline
4. Report created files

Output format:

```
  Modified src/shared/nav.ts — added Tasks nav item
  Modified src/i18n.ts — added 'tasks' namespace
  Modified src/shared/errors.ts — added tasks error group

Done. 12 files created, 4 files modified.
```
</action>

Create `bin/commands/update.ts`:

```typescript
export const updateCommand = new Command("update")
  .description("Regenerate all generated/ files from current YAML configs")
  .option("--dry-run", "Preview changes without writing")
  .option("--force", "Overwrite even if generated files have uncommitted changes")
  .action(async (options) => { ... });
```

Implementation:
1. Find all feature YAML files in the project
2. Check for uncommitted changes in `generated/` directories (git status):
   - If changes found and no --force: warn and exit
   - If changes found and --force: warn and continue
3. For each feature:
   - Read and validate YAML
   - Delete existing `generated/{feature}/` directory
   - Re-scaffold from YAML
   - Re-wire shared files (idempotent — no duplicates)
4. Report: files regenerated, templates updated, any errors

Output format:
```
Updating generated code...

Scanning for feature YAML files...
  Found 5 features: tasks, projects, subtasks, workLogs, activityLogs

Regenerating...
  tasks: 12 files regenerated
  projects: 11 files regenerated
  subtasks: 10 files regenerated
  workLogs: 9 files regenerated
  activityLogs: 8 files regenerated

Update complete. 50 files regenerated across 5 features.
```

Create `bin/commands/init.ts`:

```typescript
export const initCommand = new Command("init")
  .description("Initialize Feather project structure")
  .option("--name <name>", "Project name")
  .action(async (options) => { ... });
```

Implementation:
1. Create root `feather.yaml` if it doesn't exist (with project name from --name or prompt)
2. Create `src/generated/` directory
3. Create `src/custom/` directory
4. Create `convex/generated/` directory
5. Create `convex/custom/` directory
6. Add `generated/` to `.gitignore` if not already there (WAIT — generated code SHOULD be committed for deployments. Skip .gitignore.)
7. Report what was created
</action>
