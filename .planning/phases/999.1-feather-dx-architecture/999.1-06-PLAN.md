# Plan 999.1-06: Generator Bug Fixes and Migration Validation

## Objective

Fix the 6 consistent bugs found during generator stress tests and build the migration path for existing features (tasks, projects, subtasks, workLogs, activityLogs) from current structure to the generated/custom split. Validate that migrated features produce identical behavior.

## Why This Matters

The stress tests found 6 blocking bugs that affect every generated feature. These must be fixed before the new pipeline can produce reliable output. Additionally, 5 existing features need a migration path to the new structure — otherwise the framework only works for new features, not the existing CalmDo codebase.

## must_haves

- [ ] Bug 1 fixed: Schema enum collisions between features resolved (namespace enums by feature name)
- [ ] Bug 2 fixed: belongs_to FK columns added to schema table definition
- [ ] Bug 3 fixed: Status transitions handle branching (not just linear currentIndex + 1)
- [ ] Bug 4 fixed: FilterBar no longer generates duplicate tabs
- [ ] Bug 5 fixed: Route template uses correct auth pattern for this codebase
- [ ] Bug 6 fixed: Cross-entity features generate related record panels (addressed by Plan 04)
- [ ] Migration script moves existing feature code to generated/custom split
- [ ] Migration preserves all existing tests and behavior
- [ ] Regression tests verify each bug fix
- [ ] `npm test` passes at 100% coverage

<action>
**Bug 1: Schema enum collisions**

Problem: If `tasks` has `status: [todo, in_progress, done]` and `projects` has `status: [active, on_hold, completed, archived]`, both generate a `statusValues` constant that collides.

Fix in `templates/feature/schema.ts.hbs`:
- Namespace all generated constants by feature name: `taskStatusValues`, `projectStatusValues`
- Update all references in the template to use the namespaced name
- Export the namespaced constant for use by other templates

Before: `export const statusValues = ["todo", "in_progress", "done"] as const;`
After: `export const {{camelCase name}}StatusValues = ["todo", "in_progress", "done"] as const;`

Apply same fix to all enum fields, not just status.

**Bug 2: belongs_to FK columns missing**

Problem: When a feature YAML declares `relationships: { project: { type: belongs_to, target: projects, column: projectId } }`, the generated schema table definition does not include `projectId: v.optional(v.id("projects"))`.

Fix in `templates/feature/schema.ts.hbs`:
- After rendering regular fields, iterate over `relationships` where `type === "belongs_to"`
- For each belongs_to relationship, add the FK column to the table definition:
  ```
  {{#each relationships}}
  {{#if (eq this.type "belongs_to")}}
    {{this.column}}: v.optional(v.id("{{this.target}}")),
  {{/if}}
  {{/each}}
  ```
- Respect `required` flag: `required: true` → `v.id("target")`, `required: false` → `v.optional(v.id("target"))`
</action>
<acceptance_criteria>
- Two features with identically named enum fields (e.g., both have `status`) produce different constant names: `taskStatusValues` vs `projectStatusValues`
- `belongs_to` relationships produce FK columns in the schema table definition
- FK column uses `v.optional(v.id("target"))` for optional relationships
- FK column uses `v.id("target")` for required relationships
- Templates still produce valid TypeScript
- `npm test` passes
</acceptance_criteria>
</task>

<action>
**Bug 3: Branching transitions broken**

Problem: The status transition logic uses `currentIndex + 1` to determine the next valid state, which only works for linear sequences (todo -> in_progress -> done). Branching transitions like `open -> [in_progress, won, lost]` break.

Fix in `templates/feature/mutations.ts.hbs`:
- Replace `currentIndex + 1` logic with a transitions map lookup:
  ```typescript
  const transitions: Record<string, string[]> = {
    {{#each statusField.transitions}}
    "{{@key}}": [{{#each this}}"{{this}}"{{#unless @last}}, {{/unless}}{{/each}}],
    {{/each}}
  };

  const allowedNext = transitions[currentStatus] || [];
  if (!allowedNext.includes(newStatus)) {
    throw new Error(`Cannot transition from ${currentStatus} to ${newStatus}. Allowed: ${allowedNext.join(", ")}`);
  }
  ```
- Remove the `currentIndex + 1` assumption entirely
- Support multiple valid transitions from a single state

**Bug 4: FilterBar duplicate tabs**

Problem: When a feature has filterable enum fields AND filteredViews, the FilterBar generates tabs for both, creating duplicates.

Fix in `templates/feature/filter-bar.tsx.hbs`:
- Deduplicate: if a filteredView exists for a filterable field's values, use the filteredView (it has custom labels/filters)
- Only auto-generate tabs for filterable fields that don't have explicit filteredViews
- Add `{{#unless (hasFilteredView @key)}}` guard around auto-generated tabs
- Register a Handlebars helper `hasFilteredView` that checks if the field name matches any filteredView filter key
</action>
<acceptance_criteria>
- Status transition with `open: [in_progress, won, lost]` correctly allows all 3 transitions
- Status transition from `open` to an invalid state (e.g., `archived`) throws error
- Linear transitions (todo -> in_progress -> done) still work correctly
- FilterBar with both `filterable: true` on status field AND `filteredViews` for status does not produce duplicate tabs
- FilterBar with only `filterable: true` (no filteredViews) still generates tabs
- Templates produce valid TypeScript
- `npm test` passes
</acceptance_criteria>
</task>

Create `templates/migration/migrate-feature.ts`:

```typescript
export interface MigrationOptions {
  featureName: string;
  projectRoot: string;
  dryRun?: boolean;
}

export interface MigrationResult {
  generatedFiles: string[];     // files moved to generated/
  customFiles: string[];        // files kept in custom/
  unchangedFiles: string[];     // files that stayed in place
  warnings: string[];           // potential issues
}

export async function migrateFeature(options: MigrationOptions): Promise<MigrationResult>
```

Migration strategy:
1. Read the feature's YAML file
2. Regenerate what the pipeline would produce into a temp directory
3. Compare generated output with existing files:
   - Files that match generated output → move to `generated/{feature}/`
   - Files that differ from generated output → keep in `custom/{feature}/` as overrides
   - Update YAML `overrides` section to point to custom files
4. Update import paths in custom files: `@/features/{name}/` → `@/generated/{name}/`
5. Keep existing tests alongside their source (move test to generated/ or custom/ with its file)

Create `templates/migration/__tests__/migrate-feature.test.ts`:

- Test: migrate feature with customized component → component in custom/, override in YAML
