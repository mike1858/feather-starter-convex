# Plan 999.1-04: Cross-Entity UI Generation — Master-Detail Views and Related Records

## Objective

Build generator support for cross-entity UI — the #1 gap identified by stress tests. Every ERP-like feature needs master-detail views (orders showing line items, contacts showing deals, tasks showing subtasks + work logs + activity timeline). The generator currently has zero support for relationship-level UI. This plan adds it.

## Why This Matters

The generator stress tests showed individual entity coverage at ~65-75% but zero cross-entity support. For the user's domain (ERP-like transactional systems — orders, POS, help desk, payroll), every feature needs master-detail views. Without this, every feature requires manual UI work that the generator should handle.

## must_haves

- [ ] Detail page template renders related records panels based on YAML `detailView.relatedRecords` config
- [ ] 4 display modes for related records: table, checklist, timeline, cards
- [ ] Inline editing support for related records (create/edit without navigating away)
- [ ] Resolver reads relationship YAML config and produces template context for detail page
- [ ] Generated detail page composes from panel primitives (not monolithic template per combination)
- [ ] Tests verify detail page generation with 1, 2, and 3 related record panels
- [ ] `npm test` passes at 100% coverage

<action>
Create `templates/cross-entity/resolver.ts`:

```typescript
export interface ResolvedRelatedRecord {
  source: string;           // entity name: "subtasks"
  sourcePlural: string;     // "Subtasks"
  display: "table" | "checklist" | "timeline" | "cards";
  inline: boolean;
  readonly: boolean;
  columns: string[];        // resolved from source entity's fields
  foreignKey: string;       // resolved FK column name
  queryName: string;        // Convex query function name: "subtasks.queries.listByTask"
  mutationNames: {          // Convex mutation names for CRUD on related entity
    create?: string;
    update?: string;
    remove?: string;
  };
}

export interface DetailPageContext {
  parentEntity: string;     // "tasks"
  parentLabel: string;      // "Task"
  layout: "tabs" | "panels" | "accordion";
  relatedRecords: ResolvedRelatedRecord[];
  hasInlineEditing: boolean;  // true if any panel has inline: true
}

export function resolveDetailPage(
  parentYaml: FeatureConfig,
  relatedYamls: Map<string, FeatureConfig>  // source entity name -> its YAML config
): DetailPageContext
```

Implementation:
1. Read `detailView.relatedRecords` from parent YAML
2. For each related record:
   - Look up the source entity's YAML in relatedYamls map
   - Resolve columns from source entity's fields (use first 4-5 fields if not specified)
   - Resolve FK column from relationship definition (belongs_to.column or children foreign key)
   - Generate Convex query/mutation function names following project conventions
3. Determine layout from YAML config (default: "panels")
4. Return structured context for template rendering
</action>

- Layout defaults to "panels" when not specified

   ```tsx
   {{#if detailView.relatedRecords}}
   <div className="mt-8 space-y-6">
     {{#each detailView.relatedRecords}}
     <RelatedRecordPanel
       title="{{this.sourcePlural}}"
       display="{{this.display}}"
       data={ {{this.source}}Data }
       loading={ {{this.source}}Loading }
       {{#unless this.readonly}}
       onAdd={() => handleAdd{{this.sourcePascal}}()}
       onEdit={(id) => handleEdit{{this.sourcePascal}}(id)}
       onDelete={(id) => handleDelete{{this.sourcePascal}}(id)}
       {{/unless}}
       {{#if this.readonly}}readonly{{/if}}
       {{#if this.columns}}columns={ {{{json this.columns}}} }{{/if}}
     />
     {{/each}}
   </div>
   {{/if}}
   ```

3. Generate Convex query hooks for each related record:
   ```tsx
   {{#each detailView.relatedRecords}}
   const { data: {{this.source}}Data, isLoading: {{this.source}}Loading } =
     useQuery(api.{{this.source}}.queries.listBy{{../PascalName}}, { {{../name}}Id: id });
   {{/each}}
   ```

4. Generate mutation hooks for inline editing panels:
   ```tsx
   {{#each detailView.relatedRecords}}
   {{#if this.inline}}
   const create{{this.sourcePascal}} = useMutation(api.{{this.source}}.mutations.create);
   const update{{this.sourcePascal}} = useMutation(api.{{this.source}}.mutations.update);
   const remove{{this.sourcePascal}} = useMutation(api.{{this.source}}.mutations.remove);
   {{/if}}
   {{/each}}
   ```

5. When no `detailView.relatedRecords`, the detail page renders as before (just parent entity fields).

6. Add imports for RelatedRecordPanel component and Convex hooks.
</action>
