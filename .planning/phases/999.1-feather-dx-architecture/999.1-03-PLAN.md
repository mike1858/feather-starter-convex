# Plan 999.1-03: Generated/Custom Directory Split — Retarget Templates and Pipeline

## Objective

Implement the generated/custom code boundary (Architecture Decision Option C). Retarget Plop templates to output into `src/generated/{feature}/` and `convex/generated/{feature}/` directories. Build the two-phase pipeline: Phase 1 (Plop scaffolds into generated/), Phase 2 (ts-morph wires into shared files). Existing features continue working as-is (hybrid mode).

## Why This Matters

This is the core of safe upstream updates (D-11): generated code is always replaceable, custom code is always safe. Without this split, `feather update` can't regenerate files without destroying user customizations. The generated/custom boundary is the single most important architectural decision in the framework.

## must_haves

- [ ] Templates output to `src/generated/{feature}/` instead of `src/features/{feature}/`
- [ ] Backend templates output to `convex/generated/{feature}/` instead of `convex/{feature}/`
- [ ] Generated files include `// @generated — DO NOT EDIT` header comment
- [ ] Two-phase pipeline: scaffold (Plop) → wire (ts-morph) runs as a single command
- [ ] Pipeline reads feature YAML, validates with Zod schema, scaffolds, then wires
- [ ] Existing features in `src/features/` and `convex/` continue working (hybrid mode)
- [ ] Route files still output to `src/routes/` (TanStack Router requirement) but are thin wrappers
- [ ] Pipeline is testable — integration test generates a test feature and verifies all files created
- [ ] `npm test` passes at 100% coverage

<type>code</type>

- templates/schema/yaml-validator.ts (YAML validation from Plan 01)

</read_first>
<action>
Create `templates/pipeline/generate.ts` — the main pipeline entry point:

```typescript
export interface GenerateOptions {
  yamlPath: string;           // path to feature YAML
  projectRoot: string;        // project root directory
  outputMode: "generated" | "legacy";  // generated/ split or legacy features/ path
  dryRun?: boolean;           // preview without writing files
}

export async function generateFeature(options: GenerateOptions): Promise<GenerateResult>
```

Implementation:
1. Read and validate feature YAML (using yaml-validator from Plan 01)
2. Merge with defaults (using defaults-merger from Plan 01)
3. Call scaffold phase (Plop — creates files)
4. Call wire phase (ts-morph — modifies shared files)
5. Return result with list of files created and modifications made

Create `templates/pipeline/scaffold.ts`:
```typescript
export async function scaffoldFeature(config: FeatureConfig, options: ScaffoldOptions): Promise<string[]>
```
- Wraps Plop programmatic API (nodePlop)
- Targets output to `src/generated/{name}/` and `convex/generated/{name}/`
- Returns list of created files

Create `templates/pipeline/wire.ts`:
```typescript
export async function wireFeature(config: FeatureConfig, projectRoot: string): Promise<WiringResult>
```
- Calls `wireFeature` from Plan 02's wiring layer
- Derives wiring config from the validated YAML config
- Maps YAML fields to wiring parameters (icon from views config, path from name, errors from error defaults)

Create `templates/pipeline/index.ts` barrel export.
</action>
