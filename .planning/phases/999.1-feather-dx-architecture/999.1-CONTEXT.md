<domain>
## Phase Boundary

Evolve Feather from a starter kit to a framework with: declarative YAML config (`feather.yaml`), generated/custom code split, LLM architect conversation (entity discovery + 8-dimension behavior overlay), feature/bundle/custom composability, safe upstream updates, and telemetry.

This phase covers **design and implementation of the framework evolution**. The 8-dimension "Beyond CRUD" framework, meta-model (The Quad), and architecture (Option C — Declarative Config + Generated/Custom Split) are already decided in `.planning/design/FEATHER-DX-ARCHITECTURE.md`.

<decisions>
## Implementation Decisions

### feather.yaml Schema Organization
- **D-01:** Root `feather.yaml` for project-level config (branding, bundles, shared settings) + per-feature YAML files (one per feature, like today's `.gen.yaml` but upgraded). Features are self-contained, scales well, clean git diffs.
- **D-02:** Tight parent-child entities (PO header + line items, where child doesn't make sense without parent) are defined **inline in the parent YAML** under a `children:` section. One YAML = one installable unit.
- **D-03:** Loose associations (task optionally belongs to project) remain as today: `belongs_to` in the child YAML with `required: false`. Child exists independently.
- **D-04:** Parent YAML declares its detail view panels: `detailView: { relatedRecords: [{ source: line-items, display: table }, { source: activity-logs, display: timeline }] }`. One file tells you everything the detail page shows.
- **D-05:** Derived data / aggregations declared in the **displaying entity** (project.yaml declares `derivedData: { taskCount: count(tasks), tasksByStatus: group(tasks, status) }`). Matches Glide's computed columns pattern — declared where displayed.

### Generated/Custom Code Boundary
- **D-06:** Both mechanisms — **import direction for UI composition** (custom imports from generated, wraps/extends) + **extension points for behavior hooks** (beforeSave, onStatusChange, afterDelete declared in YAML). Matches Frappe's model: hooks.py for behavior, custom pages for UI.
- **D-07:** Generated code NEVER imports from custom. One-way dependency: custom → generated.
- **D-08:** Escape hatch: **YAML override pointer** — `overrides: { form: custom/tasks/CustomForm.tsx }` makes it explicit in YAML what's customized. Generator knows what's overridden and can warn about breaking changes. (Alternatives documented: shadow file convention, copy-eject command.)

### Quick Entry & Identity
- **D-09:** Quick entry form: **auto from required fields** by default (every feature gets it), with YAML override when needed (`form: { quickEntry: [title, priority] }`). Matches 3-tier pattern: auto-generated default, declarable override.
- **D-10:** Identity/Naming: **predefined templates + custom format**. YAML: `identity: { type: expression, format: "{PREFIX}-{YYYY}-{###}" }`. Templates: auto-increment, uuid, expression-based. Falls back to auto-increment UUID if not specified.

### Safe Upstream Updates
- **D-11:** `feather update` **hard overwrites** all `generated/` files. No warnings, no diffs. Generated = hands off. If you want to customize, use the override mechanism (D-08). Users learn the boundary fast.
- **D-12:** Breaking changes in custom imports fixed via **AST codemods** shipped with each Feather release. Like Next.js codemods — auto-fix custom files that import changed generated APIs.

### Core Design Principle
- **D-13:** **Sensible defaults, override whenever possible. Infer from context — never ask what the system can figure out.** The reference model is Tableau: drag Excel with one tab → auto-import (no "which tab?" dialog). Double-click dimension → bar chart (no chart picker wizard). Infer types from data, auto-select when there's only one option, progressive disclosure for everything. The anti-model is Power BI / SAP: name this, pick source, name that, configure columns → "configuration death." Feather must be Tableau, not Power BI. One config surface (YAML), no multi-step wizards, no right-click config panels. This maps to the 3-tier model (auto-generated → declarable → custom). Apply BEFORE asking the user — only escalate when genuinely ambiguous.

### Claude's Discretion
- **Escape hatch mechanism (D-08):** YAML override pointer chosen as best balance of explicitness and generator awareness. If this proves too verbose in practice, shadow file convention (same filename in custom/ automatically replaces generated/) is the fallback — simpler but less visible.
- **AST wiring library:** ts-morph vs jscodeshift vs custom — defer to research phase to evaluate based on Convex/React-specific constraints.
- **Telemetry specifics:** Both build-tool telemetry + one-click error reporting (decided in design doc). Implementation details deferred to planning.
- **Bundle distribution:** Deferred (decided in design doc). Focus on single-feature generation first.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **Current YAML schema** (`tasks.gen.yaml`): Already has fields, behaviors, access, views, relationships, indexes. Foundation for the expanded feature YAML — extend rather than replace.
- **`defaults.yaml`**: Defines sensible defaults for all YAML sections. Pattern to keep: convention over configuration.
- **26 Handlebars templates** (`templates/feature/`): Card view, list view, table view, detail page, form, status badge, filter bar, etc. These become the `generated/` output templates.
- **6 Zod schemas** (`src/shared/schemas/`): Tasks, projects, subtasks, work-logs, activity-logs, username. Pattern: Zod as single source of truth → `zodToConvex()` derives Convex validators.
- **`plopfile.js`**: Current generator orchestration. Evolves into `feather generate` internals.

### Established Patterns
- **Feature-folder architecture**: `src/features/{name}/` + `convex/{name}/` — the natural split point for `generated/` vs `custom/`.
- **`zCustomMutation` with Zod validation**: Backend pattern for user-input mutations. Generator already produces this.
- **Append-only extension points**: `src/shared/nav.ts`, `src/shared/errors.ts`, `src/i18n.ts` — these are the wiring files that AST-based wiring needs to modify.
- **`belongs_to` with required flag**: Existing relationship pattern (`required: true` = tight, `required: false` = loose). D-02/D-03 extend this.

### Integration Points
- **schema.ts wiring**: Adding new entities to `convex/schema.ts` — currently manual, target for AST wiring.
- **Nav + i18n + errors wiring**: Three append-only shared files that AST wiring modifies on `feather generate`.
- **Route registration**: TanStack Router file-based routes in `src/routes/` — generated by Plop templates today.

</code_context>

<specifics>
## Specific Ideas

- **Frappe as the model to follow** — adopt Frappe patterns wholesale, fix only what's broken. Same for Glide. Don't reinvent. (Memory: `feedback_frappe_model.md`)
- **Audit trail + period close, NOT Submit/Cancel/Amend** — documents freely editable, every change tracked, period-close locks records. Status flow handles business workflow, auditability is orthogonal. (Memory: `project_audit_trail_over_immutability.md`)
- **The Quad meta-model**: Data + Layout + Actions + Reports. Operations (first 3) and Analytics (Reports) are two sides of the system coin.
- **Excel is first-class** — #1 differentiator is schema reconciliation (fuzzy column rename detection, persistent column mapping, import history tracking). Build after YAML + generator fixes.
- **User domain**: ERP-like transactional systems — order management, POS, help desk, LMS, payroll. Transaction-heavy, data-heavy, approval workflows, audit trails. (Memory: `user_domain_focus.md`)
- **Cross-entity UI is the #1 gap**: Generator handles individual entities at ~65-75% but has zero support for relationship-level UI. Every ERP feature needs master-detail views.
- **Immediate next step decided**: Design the full `feather.yaml` schema format — the YAML format is the foundation everything else builds on.

</specifics>
