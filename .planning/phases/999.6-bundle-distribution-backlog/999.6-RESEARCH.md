# Phase 999.6: Bundle Distribution - Research

**Researched:** 2026-04-01
**Domain:** CLI distribution system (bundle manifest, dependency resolution, local/remote registry)
**Confidence:** HIGH

## Summary

Phase 999.6 extends the existing `feather add/remove` CLI commands to support curated multi-feature bundles with topological dependency ordering and a hybrid local-first + GitHub registry distribution model. The existing codebase provides a solid foundation: `bin/commands/add.ts` and `bin/commands/remove.ts` already handle single-feature install/remove with manifest-driven file copying; `templates/examples/` contains 3 bundled features (todos, tickets, contacts) with `manifest.json` per feature; and the `feather.yaml` root config already has `bundles: z.array(z.string()).default([])` in its schema. The `update` command already regenerates all generated code from YAML.

The primary work is: (1) rename `templates/examples/` to `templates/features/` and create `templates/bundles/` with `bundle.json` manifests, (2) add a `list` command, (3) extend `add` to resolve bundles and install features in topological order derived from `belongs_to` relationships in `feather.yaml`, (4) extend `remove` to handle bundle-level removal, (5) add GitHub raw content fetching as a fallback when local templates are not found, and (6) extend `update` to optionally sync from a remote registry before regenerating.

**Primary recommendation:** Build incrementally: rename + bundle manifest first, then dependency resolution, then `list` command, then remote registry support. Each layer builds on the previous and is independently testable.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Two concepts only -- **features** (atomic installable unit) and **bundles** (named collection of features). "Example" is a documentation label, not a CLI concept. `templates/examples/` renamed to `templates/features/`.
- **D-02:** Bundle manifest is `bundle.json` containing feature list + metadata only: `{ name, label, description, complexity, features: [...] }`. No `installOrder` -- derived from `feather.yaml` dependency graph. No cross-entity wiring -- that stays in individual `feather.yaml` `detailView.relatedRecords` declarations.
- **D-03:** `templates/bundles/` directory, separate from `templates/features/`. Auto-detect in `feather add` checks `bundles/` first, then `features/`.
- **D-04:** Local-first with GitHub fallback. `feather add <name>` checks local `templates/` first. If not found, fetches from registry URL configured in `feather.yaml` (`registry.url`). Downloaded bundles/features cached locally in `templates/` for offline use.
- **D-05:** Registry is the GitHub repo itself -- features and bundles served as raw files from the repo. No custom infrastructure. Works with private repos (token auth). Versioning via branches/tags in registry config.
- **D-06:** `bundle.json` format is registry-compatible -- identical whether read from filesystem or fetched from URL. No format changes needed when adding remote support.
- **D-07:** Simple verbs, no double-dash flags. Full command set: `feather add <name>`, `feather remove <name>`, `feather list`, `feather update`, `feather update <name>`.
- **D-08:** `feather update` without registry configured regenerates from local templates only (backward compatible). With registry configured, syncs first then regenerates.
- **D-09:** Topological sort derived from `belongs_to` declarations in each feature's `feather.yaml`. Builds DAG, produces install order. ~20 lines of code.
- **D-10:** Already-installed features get updated (overwrite generated/, preserve custom/), missing features get installed. No skipping.
- **D-11:** Missing dependency outside the bundle = error with clear message.

### Pattern 1: Bundle Manifest Format (D-02)
**What:** A `bundle.json` file in `templates/bundles/{name}/` that declares which features comprise the bundle.
**When to use:** Every bundle directory must have one.
**Example:**
```json
{
  "name": "project-management",
  "label": "Project Management",
  "description": "Tasks, projects, subtasks, work logs, and activity tracking",
  "complexity": "advanced",
  "features": ["tasks", "projects", "subtasks", "work-logs", "activity-logs"]
}
```

### Pattern 2: Auto-Detection Resolution (D-03)
**What:** `feather add <name>` checks `templates/bundles/{name}/bundle.json` first, then `templates/features/{name}/manifest.json`. If neither found locally, falls back to registry (D-04).
**When to use:** Every `add` invocation.
**Example:**
```typescript
// Source: D-03, D-04 from CONTEXT.md
function resolve(name: string, projectRoot: string): Resolution {
  const bundlePath = path.join(projectRoot, "templates/bundles", name, "bundle.json");
  if (fs.existsSync(bundlePath)) {
    return { type: "bundle", path: bundlePath, manifest: JSON.parse(fs.readFileSync(bundlePath, "utf-8")) };
  }
  const featurePath = path.join(projectRoot, "templates/features", name, "manifest.json");
  if (fs.existsSync(featurePath)) {
    return { type: "feature", path: featurePath, manifest: JSON.parse(fs.readFileSync(featurePath, "utf-8")) };
  }
  // Fall back to remote registry (D-04)
  return { type: "not-found", name };
}
```

### Pattern 3: Topological Sort from belongs_to (D-09)
**What:** Read each feature's `feather.yaml`, extract `relationships.*.type === "belongs_to"` entries, build a DAG, and sort.
**When to use:** Before installing a bundle's features.
**Example:**
```typescript
// Source: D-09 from CONTEXT.md, ~20 lines
interface DepGraph { [feature: string]: string[] }

function buildGraph(features: string[], templatesDir: string): DepGraph {
  const graph: DepGraph = {};
  for (const name of features) {
    const yamlPath = path.join(templatesDir, name, "frontend", "feather.yaml");
    if (!fs.existsSync(yamlPath)) { graph[name] = []; continue; }
    const parsed = YAML.parse(fs.readFileSync(yamlPath, "utf-8"));
    const deps: string[] = [];
    if (parsed.relationships) {
      for (const rel of Object.values(parsed.relationships)) {
        if ((rel as any).type === "belongs_to") deps.push((rel as any).target);
      }
    }
    graph[name] = deps;
  }
  return graph;
}

function topoSort(graph: DepGraph): string[] {
  const visited = new Set<string>();
  const result: string[] = [];
  function visit(node: string) {
    if (visited.has(node)) return;
    visited.add(node);
    for (const dep of graph[node] ?? []) visit(dep);
    result.push(node);
  }
  for (const node of Object.keys(graph)) visit(node);
  return result;
}
```

### Pattern 4: Registry Fetch (D-04, D-05)
**What:** When local resolution fails, fetch from GitHub using the raw content API. Cache downloaded files in `templates/` for offline use.
**When to use:** When `feather add <name>` can't find the name locally and a registry URL is configured.
**Example:**
```typescript
// Source: D-04, D-05 from CONTEXT.md
// Registry URL format: https://raw.githubusercontent.com/{owner}/{repo}/{ref}/templates/
// Configured in feather.yaml: registry: { url: "https://raw.githubusercontent.com/..." }
// Auth for private repos: FEATHER_REGISTRY_TOKEN env var → Authorization: Bearer header

async function fetchFromRegistry(
  name: string, registryUrl: string, token?: string
): Promise<BundleOrFeature | null> {
  const headers: Record<string, string> = {};
  if (token) headers["Authorization"] = `Bearer ${token}`;

  // Try bundle first
  const bundleUrl = `${registryUrl}/bundles/${name}/bundle.json`;
  const bundleRes = await fetch(bundleUrl, { headers });
  if (bundleRes.ok) {
    return { type: "bundle", manifest: await bundleRes.json() };
  }

  // Try feature
  const featureUrl = `${registryUrl}/features/${name}/manifest.json`;
  const featureRes = await fetch(featureUrl, { headers });
  if (featureRes.ok) {
    return { type: "feature", manifest: await featureRes.json() };
  }

  return null;
}
```

### Anti-Patterns to Avoid
- **Explicit installOrder in bundle.json:** D-09 explicitly rejects this. Install order MUST come from `feather.yaml` dependency graph, never from the bundle manifest. An explicit list can get out of sync.
- **Cross-entity wiring in bundle.json:** D-02 explicitly rejects this. Wiring stays in individual `feather.yaml` `detailView.relatedRecords` declarations.
- **Global CLI install:** D-10 from Phase 999.4 mandates project-local CLI only (`npx feather`). No global install pattern.
- **Mixing v.id() with Zod in mutations:** Existing project gotcha documented in CLAUDE.md.

## Common Pitfalls

### Pitfall 1: Stale Path References After examples/ -> features/ Rename
**What goes wrong:** `add.ts`, `remove.ts`, `bundle-examples.ts`, and all tests hardcode `templates/examples/` paths. After rename, everything breaks.
**Why it happens:** The path `templates/examples` appears in at least 5 files: `add.ts` (findTemplatesDir), `remove.ts` (findTemplatesDir), `bundle-examples.ts` (output paths), `add.test.ts` (test fixtures), `remove.test.ts` (test fixtures).
**How to avoid:** Rename as the very first task. Update all path references in a single commit. Use `Grep` to find ALL occurrences of `templates/examples` before committing.
**Warning signs:** Tests fail with "not found" errors after rename.

### Pitfall 2: Circular Dependencies in belongs_to Graph
**What goes wrong:** If two features declare `belongs_to` each other (shouldn't happen but could in a malformed YAML), topological sort enters infinite recursion.
**Why it happens:** DFS-based topoSort without cycle detection.
**How to avoid:** Track "currently visiting" nodes in addition to "visited" nodes. If a node is encountered while already being visited, that's a cycle -- throw a descriptive error.
**Warning signs:** CLI hangs or stack overflow during `feather add <bundle>`.

### Pitfall 3: Remote Fetch Downloading Incomplete Feature
**What goes wrong:** A feature has multiple files (frontend/, backend/, schema/, route/, locales/). Fetching just `manifest.json` from the registry isn't enough -- need all files.
**Why it happens:** GitHub raw content API serves individual files, not directories.
**How to avoid:** Two approaches: (a) the registry serves a pre-packaged tar/zip of each feature, or (b) the `manifest.json` lists all files and the CLI fetches each one. Option (b) is simpler and consistent with D-06 (format identical local and remote). The manifest already has a `files` object listing all paths.
**Warning signs:** Feature installed but missing files (partial install).

### Pitfall 4: feather.yaml Location Inconsistency
**What goes wrong:** Feature YAMLs are found at `src/features/{name}/feather.yaml` in the project but at `templates/features/{name}/frontend/feather.yaml` in bundled templates. The topological sort needs to read the right path depending on context.
**Why it happens:** The bundling script (`bundle-examples.ts`) copies from `src/features/` into `templates/examples/{name}/frontend/`, so the YAML ends up nested under `frontend/`.
**How to avoid:** Normalize YAML lookup: check both `templates/features/{name}/feather.yaml` (top-level) and `templates/features/{name}/frontend/feather.yaml` (current bundled location). Or restructure bundling to put `feather.yaml` at the feature root.
**Warning signs:** Dependency resolution returns empty graph (no relationships found).

### Pitfall 5: Dependencies Outside the Bundle
**What goes wrong:** A bundle like `project-management` includes `subtasks` which `belongs_to` `tasks`. If `tasks` isn't in the bundle AND isn't installed, install should fail clearly (D-11).
**Why it happens:** The topoSort includes all features in the graph but doesn't check if external dependencies exist in the project.
**How to avoid:** After building the dependency graph, verify every referenced target is either (a) in the bundle's feature list, or (b) already installed in the project. If neither, fail with the message from D-11.
**Warning signs:** Features installed without their parents, leading to runtime errors.

## Open Questions

1. **Where does feather.yaml live in bundled features?**
   - What we know: In the project, it's at `src/features/{name}/feather.yaml`. In bundled templates, it's at `templates/examples/{name}/frontend/feather.yaml` (inside the frontend subdirectory).
   - What's unclear: Should the bundle script copy it to the feature root (`templates/features/{name}/feather.yaml`) for consistency, or continue nesting under `frontend/`?
   - Recommendation: Copy to feature root for clean resolution. The topo-sort needs to find it reliably, and having a canonical location (`templates/features/{name}/feather.yaml`) is simpler than searching multiple paths.

2. **Manifest file structure for remote fetch**
   - What we know: Locally, each feature has a `manifest.json` listing file paths. The GitHub raw API serves individual files.
   - What's unclear: How to fetch an entire feature directory tree from GitHub without listing every file.
   - Recommendation: Extend `manifest.json` to include a flat list of ALL files (not just category paths). The CLI fetches each file individually. This keeps the format identical local/remote per D-06. Alternative: add a `files_flat` array to manifest.json listing every individual file path.

3. **How `feather update` interacts with registry sync**
   - What we know: D-08 says update without registry = regenerate only; with registry = sync + regenerate.
   - What's unclear: Should sync update the bundle/feature template files in `templates/`, or just regenerate from remote YAML directly?
   - Recommendation: Sync downloads to `templates/` first (caching), then regenerate. This ensures offline use works after one sync (D-04).
