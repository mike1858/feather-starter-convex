**Researched:** 2026-03-10
**Domain:** Git branch plugin architecture, code extraction/refactoring

## Summary

This phase extracts all Stripe/billing code from the core starter into a `plugin/billing` git branch, following the same pattern as the three existing plugins (`plugin/infra-ci-github-actions`, `plugin/ui-command-palette`, `plugin/feature-admin-panel`). The extraction is well-scoped: 22 files move entirely to the plugin, 9 files need surgical edits in core. The main complexity is not the extraction itself but the downstream impact on the 3 existing plugin branches, which all currently contain billing code because they forked from a billing-included main.

**Primary recommendation:** Execute in two waves: (1) extract billing from core to produce a clean main, then create plugin/billing from the diff; (2) rebase/update the 3 existing plugin branches onto the new billing-free main. The predev npm script (`convex dev --run init`) must be modified or removed since `init.ts` is the Stripe product seeder.

## Architecture Patterns

### Plugin Branch Pattern (Established)

The project already has 3 working plugin branches. The pattern is:

1. Plugin branch diverges from `main` with additive changes
2. Install via `scripts/plugin.sh install plugin/<name>` (runs `git merge`)
3. Extension points are append-only arrays for clean merges:
   - `src/shared/nav.ts` -- `navItems` array
   - `src/shared/errors.ts` -- `ERRORS` object
   - `src/i18n.ts` -- `ns` namespace array

**Key insight for billing plugin:** Unlike the other 3 plugins which only _add_ things, the billing plugin must also _modify_ existing files (schema.ts, users/queries.ts, onboarding/mutations.ts, etc.). This means the plugin branch's diff against main will include both new files AND file modifications -- similar to how existing plugins already modify `vitest.config.ts` and route files.

### Account Deletion Simplification

Current `deleteCurrentUserAccount` queries subscriptions table and schedules Stripe cancellation. After extraction:

```typescript
// convex/users/mutations.ts (simplified deleteCurrentUserAccount handler)
handler: async (ctx) => {
  const userId = await auth.getUserId(ctx);
  if (!userId) return;
  const user = await ctx.db.get(userId);
  if (!user) throw new Error("User not found");
  await ctx.db.delete(userId);
  // Clean up auth accounts
  await asyncMap(["resend-otp", "github", "password", "password-reset"], async (provider) => {
    const authAccount = await ctx.db
      .query("authAccounts")
      .withIndex("userIdAndProvider", (q) =>
        q.eq("userId", userId).eq("provider", provider),
      )
      .unique();
    if (!authAccount) return;
    await ctx.db.delete(authAccount._id);
  });
},
```

Note: The auth providers list should include "password" and "password-reset" which were added in Phase 2.

### getCurrentUser Simplification

```typescript
// convex/users/queries.ts (simplified)
export const getCurrentUser = query({
  args: {},
  handler: async (ctx) => {
    const userId = await auth.getUserId(ctx);
    if (!userId) return;
    const user = await ctx.db.get(userId);
    if (!user) return;
    const avatarUrl = user.imageId
      ? await ctx.storage.getUrl(user.imageId)
      : user.image;
    return { ...user, avatarUrl: avatarUrl || undefined };
  },
});
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Plugin branch creation | Manual cherry-pick workflow | Make all core changes on main, then create plugin/billing by branching from pre-extraction commit and rebasing plugin-specific additions | Cherry-picking dozens of files is error-prone |
| Existing plugin updates | Manual conflict resolution per plugin | Rebase each plugin branch onto new main | git rebase handles the mechanical work; only real conflicts need manual resolution |

## Common Pitfalls

### Pitfall 1: predev Script Breaks Without init.ts
**What goes wrong:** `package.json` has `"predev": "convex dev --run init --until-success"` which runs `convex/init.ts` (Stripe seeder). Removing init.ts without updating predev causes `npm run dev` to fail.
**How to avoid:** Either remove the predev script entirely or replace init.ts with a no-op placeholder. A no-op init is safer for the plugin pattern -- the billing plugin can replace init.ts with the Stripe seeder.
**Recommendation:** Keep init.ts as a no-op (`export default internalAction(async () => { /* no-op */ });`) so the predev script works and plugin/billing can override it.

### Pitfall 2: TypeScript Compilation After Removing Billing Types
**What goes wrong:** `PlanKey`, `CURRENCIES`, `PLANS`, `INTERVALS` are imported across many files. If any import remains after extraction, TypeScript will fail.
**How to avoid:** After removing billing exports from schema.ts, run `npx tsc -p tsconfig.app.json --noEmit` to catch all broken imports.

### Pitfall 3: Coverage Config References Deleted Files
**What goes wrong:** `vitest.config.ts` coverage exclude list references `convex/http.ts`, `convex/init.ts`, and billing-related paths. Some of these may need updating.
**How to avoid:** After extraction, `convex/http.ts` is now a trivial file (8 lines) -- it can be removed from the exclude list or kept (harmless). The billing-specific test files will be gone, so they naturally leave the coverage scope.

### Pitfall 4: Existing Plugin Branches Contain Billing Code
**What goes wrong:** All 3 existing plugin branches were forked from main when billing existed. Their diffs against main include billing code in shared files (`nav.ts`, `errors.ts`, `schema.ts`). If rebased naively, the billing code could conflict with the new billing-free main.
**Why it happens:** Plugin branches carry the full file content at their fork point. `nav.ts` in each plugin includes the Billing nav item. `errors.ts` includes the billing error group.
**How to avoid:** Rebase each plugin branch onto the new billing-free main. Conflicts will occur in the shared extension point files -- resolve by keeping the plugin's _own_ additions but dropping billing entries. Test each rebased plugin by verifying `npm test` passes after merge.

### Pitfall 5: E2E Test References Billing
**What goes wrong:** `e2e/settings.spec.ts` has a "billing page loads" test that visits `/dashboard/settings/billing`. This test will fail when the billing route is removed.
**How to avoid:** Remove the billing E2E test from core. It moves to the plugin branch.

### Pitfall 6: lefthook.yml convex typecheck exclusion
**What goes wrong:** `lefthook.yml` currently excludes `convex/tsconfig.json` from typecheck "due to pre-existing TS2554 error in convex/http.ts". After extracting billing, `convex/http.ts` becomes trivial (just auth routes). The TS2554 error likely goes away, meaning the typecheck exclusion can potentially be removed.
**How to avoid:** After extraction, test if `npx tsc -p convex/tsconfig.json --noEmit` passes. If it does, re-enable in lefthook.yml.

### Pitfall 7: getLocaleCurrency in misc.ts
**What goes wrong:** `src/utils/misc.ts` imports `CURRENCIES` from `@cvx/schema` and exports `getLocaleCurrency()`. After removing CURRENCIES from schema, this breaks.
**How to avoid:** Remove `getLocaleCurrency` and the `CURRENCIES` import from `misc.ts`. The function only existed for billing (onboarding currency selection, billing settings). It moves to the plugin.

## Open Questions

1. **dangerouslyIgnoreUnhandledErrors flag removal**
   - What we know: The flag was added because Stripe SDK instantiation in `convex/billing/stripe.ts` causes unhandled rejections when `STRIPE_SECRET_KEY` is missing in test env
   - What's unclear: Whether removing the Stripe code fully eliminates the unhandled rejections, or if something else also contributes
   - Recommendation: After extraction, test `npx vitest run --coverage` without the flag. If it passes, remove from `lefthook.yml`

2. **Plugin/billing branch creation strategy**
   - What we know: Need a branch where merging into billing-free main adds billing back
   - Options: (A) Branch from current main (pre-extraction), then the "plugin" is really the current state; (B) Branch from new billing-free main, re-add billing as commits
   - Recommendation: Option B -- branch from billing-free main and add billing files as new commits. This ensures the plugin diff is purely additive against the current main, matching how other plugins work. The billing code can be copied from the pre-extraction state.

3. **init.ts handling**
   - What we know: `predev` script runs `convex dev --run init`. init.ts seeds Stripe products.
   - Recommendation: Replace with no-op internalAction in core. Plugin/billing overrides with Stripe seeder.
