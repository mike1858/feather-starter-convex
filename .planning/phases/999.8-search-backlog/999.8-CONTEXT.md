# Phase 999.8: Search — Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can text-search across task titles and project names, and filter task/project lists by status, priority, project, and assignee. 6 requirements carried forward from Phase 6: SRCH-01 to SRCH-06.

</domain>

<decisions>
## Implementation Decisions

### Search Implementation
- **D-01:** Convex full-text search indexes (`searchIndex` in schema.ts). Native, real-time reactive, sub-millisecond, prefix matching. No external search service.
- **D-02:** Search indexes on: tasks (title, description), projects (name, description).

### Search UX
- **D-03:** Global search box in the header/nav bar. Always visible. Searches across all entities (tasks + projects). Results appear as a dropdown overlay while typing.
- **D-04:** No per-page search — the global search handles cross-entity lookup. Per-page filtering is handled by the filter bar (separate from search).

### Filter Controls
- **D-05:** Horizontal filter bar above each list page. Row of dropdown filters, all always visible. Like GitHub Issues.
- **D-06:** Task list filters: Status (todo/in_progress/done), Priority (high/normal), Assignee (me/unassigned/specific person), Project.
- **D-07:** Project list filters: Status (active/on_hold/completed/archived).
- **D-08:** Filters persist via URL query params (`/tasks?status=in_progress&assignee=me`). Shareable, bookmarkable, survives refresh.

### Claude's Discretion
- Search dropdown styling and result grouping (tasks vs projects sections)
- Debounce timing for search input (200-300ms typical)
- Filter dropdown component choice (Radix Select vs custom)
- Whether "Clear all filters" button is needed
- Keyboard shortcut for global search (Cmd+K is standard)

</decisions>

## Existing Code Insights

### Reusable Assets
- **`templates/feature/filter-bar.tsx.hbs`**: Handlebars template for filter bars — can reference for pattern but will be hand-built for search-specific needs.
- **Radix UI primitives in `src/ui/`**: Select, Input, Popover components available for filter dropdowns and search overlay.
- **`convex/tasks/queries.ts`**: Existing query patterns — extend with filtered/searchable variants.

### Established Patterns
- **Convex queries with filters**: Current queries use `.filter()` on the client side. Search indexes enable server-side filtering.
- **URL-based routing**: TanStack Router supports search params natively — filter state maps directly to route search params.

### Integration Points
- **`convex/schema.ts`**: Add `searchIndex` declarations for tasks and projects tables.
- **Header/nav component**: Add global search box alongside existing nav items.
- **Task list pages**: Add filter bar component above existing list views.
- **TanStack Router search params**: Wire filter state to URL query params via route definition.

<specifics>
## Specific Ideas

- Global search in header matches Linear's pattern — one search box, searches everything, results in dropdown
- Horizontal filter bar matches GitHub Issues — familiar, simple, all filters visible
- URL params for filter persistence means team members can share filtered views ("here are the high-priority tasks assigned to you")

</specifics>
