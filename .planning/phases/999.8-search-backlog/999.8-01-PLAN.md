# Plan 999.8-01: Search Indexes, Filtered Queries, and Search Queries

## Objective

Add Convex full-text search indexes on tasks and projects tables, create filtered task query with multi-parameter filtering (status, priority, assignee, project), add text search queries for both tasks and projects, and write backend tests for all new queries.

## Context

Phase 6 built activity logs but deferred search (SRCH-01 through SRCH-06). The tasks table currently has `by_assignee`, `by_visibility`, `by_creator`, `by_assignee_status`, and `by_project` indexes. The projects table has `by_status` and `by_creator` indexes. No search indexes exist. The `projects.queries.list` already accepts an optional `status` filter — SRCH-05 is partially implemented but needs the filter bar UI (Plan 02).

## Must-Haves (Goal-Backward Verification)

- [ ] `convex/schema.ts` contains `.searchIndex("search_title", ...)` on tasks table with filterFields: status, assigneeId, projectId, priority

- [ ] `convex/tasks/queries.ts` exports a `search` query that accepts `searchTerm` and returns matching tasks

- [ ] `convex/projects/queries.ts` exports a `search` query that accepts `searchTerm` and returns matching projects
- [ ] All new queries require authentication (return empty array for unauthenticated users)
- [ ] Backend tests pass for all new queries

<action>
Add a `search` query to `convex/tasks/queries.ts`:

```typescript
export const search = query({
  args: { searchTerm: v.string(), limit: v.optional(v.number()) },
  handler: async (ctx, args) => {
    const userId = await auth.getUserId(ctx);
    if (!userId) return [];
    if (!args.searchTerm.trim()) return [];
    return ctx.db
      .query("tasks")
      .withSearchIndex("search_title", (q) =>
        q.search("title", args.searchTerm)
      )
      .take(args.limit ?? 10);
  },
});
```

Key behaviors:
- Returns empty array if not authenticated
- Returns empty array if search term is empty/whitespace
- Uses `search_title` search index
- Limits results to 10 by default (overridable via `limit` arg)
- No visibility filtering — global search shows all tasks the user has access to (scoping to org in v3.0)
</action>

<action>
Add a `listFiltered` query to `convex/tasks/queries.ts` that accepts optional filter parameters and returns filtered tasks. Import necessary validators:

```typescript
import { zodToConvex } from "convex-helpers/server/zod4";
import { taskStatus } from "../../src/shared/schemas/tasks";

export const listFiltered = query({
  args: {
    status: v.optional(zodToConvex(taskStatus)),
    priority: v.optional(v.boolean()),
    assigneeId: v.optional(v.string()),
    projectId: v.optional(v.id("projects")),
  },
  handler: async (ctx, args) => {
    const userId = await auth.getUserId(ctx);
    if (!userId) return [];

    // Start with all tasks (no single index covers all filter combinations)
    let tasks = await ctx.db.query("tasks").collect();

    // Apply filters client-side
    if (args.status !== undefined) {
      tasks = tasks.filter((t) => t.status === args.status);
    }
    if (args.priority !== undefined) {
      tasks = tasks.filter((t) => t.priority === args.priority);
    }
    if (args.assigneeId !== undefined) {
      if (args.assigneeId === "me") {
        tasks = tasks.filter((t) => t.assigneeId === userId);
      } else if (args.assigneeId === "unassigned") {
        tasks = tasks.filter((t) => !t.assigneeId);
      } else {
        tasks = tasks.filter((t) => t.assigneeId === args.assigneeId);
      }
    }
    if (args.projectId !== undefined) {
      tasks = tasks.filter((t) => t.projectId === args.projectId);
    }

    return tasks.sort((a, b) => a.position - b.position);
  },
});
```

The `assigneeId` field accepts special string values:
- `"me"` — tasks assigned to the current user
- `"unassigned"` — tasks with no assignee
- Any other string — treated as a user ID for a specific person

This approach uses client-side filtering because Convex doesn't support compound queries across multiple filter dimensions on different indexes. For the expected data volume (< 10,000 tasks), this is efficient.
</action>
