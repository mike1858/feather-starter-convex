# Phase 999.7: Telemetry - Research

**Researched:** 2026-04-01
**Domain:** CLI telemetry, error capture/reporting, Convex scheduled functions, React ErrorBoundary, Sentry integration
**Confidence:** HIGH

## Summary

Phase 999.7 adds three interconnected subsystems: (1) CLI telemetry that tracks `feather` command usage locally in `.feather/telemetry.json` with optional remote reporting, (2) a `devErrors` error capture system mirroring the existing `devEmails` pattern (Convex table + `/dev/errors` route + env var gating), and (3) a scheduled error digest via Convex cron jobs that pushes summaries to an admin Convex instance.

The existing codebase provides strong patterns to follow. The `devEmails` table, `convex/devEmails/mutations.ts`, and `src/routes/_app/dev/mailbox.tsx` serve as direct models for the `devErrors` subsystem. The CLI already uses Commander.js with a clean action/command separation pattern (`bin/commands/*.ts` with `*Action` functions for testability). Convex natively supports cron jobs via `cronJobs()` and HTTP actions via `httpRouter()` — both needed for the digest pipeline. The React 19 error hooks (`onUncaughtError`, `onCaughtError`, `onRecoverableError`) are the modern approach for error capture, replacing the older class-based ErrorBoundary. Sentry integration is opt-in Layer 2 via `@sentry/react` which provides both `Sentry.ErrorBoundary` and `Sentry.reactErrorHandler()` for React 19.

**Primary recommendation:** Build in three waves: (1) local CLI telemetry + `feather stats`, (2) `devErrors` table + ErrorBoundary + backend error capture + `/dev/errors` route, (3) scheduled digest + Sentry opt-in + admin endpoint.

### Locked Decisions
- **D-01:** Every `feather` CLI command appends to `.feather/telemetry.json` locally. Tracks: command name, feature/bundle name, success/failure, duration, timestamp.

- **D-04:** `devErrors` Convex table stores every error with full context. Same pattern as existing `devEmails` table / `/dev/mailbox` route.
- **D-05:** `/dev/errors` route inside the dashboard layout shows all captured errors. Visual dashboard — error list with expandable details (stack trace, route, function name, args, browser info, timestamp). Includes "Clear All" action.
- **D-06:** Auto-capture from 4 error sources: Frontend crash (React ErrorBoundary), Backend error (Convex function error wrapper/middleware), Silent failure (ConvexError pattern), Startup error (CLI captures to local `.feather/errors.json`).
- **D-07:** `/dev/errors` route gated by `DEV_ERRORS` env var (same pattern as `DEV_MAILBOX`). Defaults to enabled in dev.
- **D-08:** Convex scheduled function (cron) runs on configurable interval, collects error summary from `devErrors` table, and POSTs to maintainer's endpoint.
- **D-09:** Client Convex to maintainer's admin Convex instance. The admin instance aggregates error digests from all client deployments into one view.
- **D-10:** Schedule configurable: `hourly | daily | weekly`. Default: `daily`.
- **D-11:** Digest contents: error count since last report, top errors by frequency (message + stack trace + route/function), environment info. No user PII, no data values.

- **D-13:** Sentry integration as opt-in Layer 2. Activates when DSN is configured via `feather.yaml` (`errorReporting.sentry.dsn`) or `SENTRY_DSN` env var.

- **D-16:** `feather stats` CLI command reads local `.feather/telemetry.json`. Terminal output only.
- **D-17:** Three telemetry tiers: basic (default), schema (opt-in), full (opt-in, beta clients).
- **D-18:** Never collected at any tier: API keys, environment variables, secrets, auth tokens.
- **D-19:** Beta clients run `level: full`. Public users get `level: basic` by default.
- **D-20:** All telemetry configured in `feather.yaml` under a `telemetry` section.

### Pattern 1: devErrors Table (Mirror devEmails)
**What:** Convex table storing captured errors with full context
**When to use:** Every error captured from any of the 4 sources (D-06)
**Example:**
```typescript
// convex/devErrors/mutations.ts — mirrors convex/devEmails/mutations.ts
// Source: existing devEmails pattern
import { v } from "convex/values";
import { mutation } from "@cvx/_generated/server";

export const store = mutation({
  args: {
    source: v.union(
      v.literal("frontend"),
      v.literal("backend"),
      v.literal("silent"),
      v.literal("startup"),
    ),
    message: v.string(),
    stack: v.optional(v.string()),
    route: v.optional(v.string()),
    functionName: v.optional(v.string()),
    args: v.optional(v.string()),    // JSON-stringified, sanitized
    browserInfo: v.optional(v.string()),
    timestamp: v.number(),
    digested: v.optional(v.boolean()), // marks if included in a digest
  },
  handler: async (ctx, args) => {
    await ctx.db.insert("devErrors", args);
  },
});

export const clearAll = mutation({
  args: {},
  handler: async (ctx) => {
    const errors = await ctx.db.query("devErrors").collect();
    for (const error of errors) {
      await ctx.db.delete(error._id);
    }
  },
});
```

### Pattern 5: CLI Telemetry Wrapper
**What:** Wraps every CLI command to record timing, success/failure, and feature context
**When to use:** Around every `feather` CLI command action
**Example:**
```typescript
// bin/lib/telemetry.ts
import * as fs from "node:fs";
import * as path from "node:path";
import * as yaml from "yaml";

interface TelemetryEvent {
  command: string;
  feature?: string;
  bundle?: string;
  success: boolean;
  durationMs: number;
  timestamp: string;
  level?: "basic" | "schema" | "full";
}

const TELEMETRY_FILE = ".feather/telemetry.json";

export function isEnabled(projectRoot: string): boolean {
  if (process.env.FEATHER_TELEMETRY === "false") return false;
  // Read feather.yaml telemetry.enabled
  const yamlPath = path.join(projectRoot, "feather.yaml");
  if (!fs.existsSync(yamlPath)) return true; // default enabled
  const config = yaml.parse(fs.readFileSync(yamlPath, "utf-8"));
  return config?.telemetry?.enabled !== false;
}

export function appendEvent(projectRoot: string, event: TelemetryEvent): void {
  const filePath = path.join(projectRoot, TELEMETRY_FILE);
  const dir = path.dirname(filePath);
  if (!fs.existsSync(dir)) fs.mkdirSync(dir, { recursive: true });
  const events: TelemetryEvent[] = fs.existsSync(filePath)
    ? JSON.parse(fs.readFileSync(filePath, "utf-8"))
    : [];
  events.push(event);
  fs.writeFileSync(filePath, JSON.stringify(events, null, 2));
}
```

### Pattern 6: Sentry Conditional Initialization (Layer 2)
**What:** Initialize Sentry only when DSN is available; dual-report to both devErrors and Sentry
**When to use:** App startup when `SENTRY_DSN` or `VITE_SENTRY_DSN` env var is set
**Example:**
```typescript
// src/shared/error-capture.ts
import * as Sentry from "@sentry/react";

let sentryInitialized = false;

export function initErrorCapture() {
  const dsn = import.meta.env.VITE_SENTRY_DSN;
  if (dsn) {
    Sentry.init({ dsn });
    sentryInitialized = true;
  }
}

export function captureError(
  source: "frontend" | "backend" | "silent" | "startup",
  error: Error,
  context?: Record<string, unknown>,
) {
  // Always store to devErrors (Layer 1)
  storeToDevErrors(source, error, context);
  // Also report to Sentry if configured (Layer 2)
  if (sentryInitialized) {
    Sentry.captureException(error, { extra: context });
  }
}
```

### Anti-Patterns to Avoid
- **Blocking CLI on telemetry POST:** Remote telemetry should be fire-and-forget. Never let a network failure slow down or break a CLI command.
- **Storing raw args/data in devErrors:** Sanitize before storage. D-18 prohibits API keys, env vars, secrets, auth tokens. Strip sensitive fields from `args` before storing.
- **Tight coupling between error capture and Sentry:** The error capture utility must work without Sentry. Sentry is an optional enhancement, not a requirement.
- **Creating crons.ts with hardcoded schedule:** The schedule (hourly/daily/weekly) should be configurable. However, Convex crons are defined at deploy time, not runtime. Use the `@get-convex/crons` component for runtime-configurable cron schedules, or accept that schedule changes require redeployment.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cron scheduling | Custom interval timer | Convex `cronJobs()` | Native, managed, dashboard-visible, at-most-once guarantee |
| React error capture | Class-based ErrorBoundary | React 19 `onUncaughtError`/`onCaughtError` hooks | React 19 hooks capture errors at root level without class component boilerplate |
| Production error monitoring | Custom error aggregation | Sentry `@sentry/react` | Industry standard; source maps, stack traces, breadcrumbs, grouping — years of edge case handling |
| HTTP endpoint auth | Custom token validation | Convex HTTP actions with shared secret header | Simple `Authorization: Bearer <shared-secret>` check in the httpAction handler |

**Key insight:** The devEmails pattern already proves the "Convex table + route + env var gating" approach works well. Replicating it for devErrors is straightforward. The complexity lives in the error capture middleware (4 sources) and the digest pipeline (cron + HTTP action + admin aggregation).

## Common Pitfalls

### Pitfall 1: Convex Cron Schedule is Deploy-Time, Not Runtime
**What goes wrong:** Developer configures `schedule: weekly` in feather.yaml, expects it to change the cron without redeployment.
**Why it happens:** Convex `cronJobs()` is evaluated at deploy time. The schedule is baked into the deployment.
**How to avoid:** Either accept deploy-time configuration (simplest) or use the `@get-convex/crons` component for runtime-configurable schedules. For an MVP, deploy-time is fine — document that schedule changes require `npx convex deploy`.
**Warning signs:** User changes feather.yaml schedule but digest timing doesn't change.

### Pitfall 2: ConvexHttpClient in Error Capture (Same as devEmails)
**What goes wrong:** Frontend error capture tries to store to devErrors via ConvexHttpClient, but the Convex URL isn't available or the mutation fails silently.
**Why it happens:** The existing devEmails pattern uses `ConvexHttpClient` from auth providers (server-side context). Frontend error capture should use the already-connected `ConvexReactClient` mutation instead.
**How to avoid:** Use `useMutation` hook for frontend error capture (inside React tree). For capture outside React tree (e.g., `onUncaughtError`), use a module-level reference to the Convex client.
**Warning signs:** Errors captured in console but not appearing in `/dev/errors`.

### Pitfall 3: Circular Error — Error Capture Throws Error
**What goes wrong:** The error capture mutation itself fails (e.g., validation error, Convex down), causing another error, creating an infinite loop.
**Why it happens:** No guard against error capture failures.
**How to avoid:** Wrap the capture call in try/catch that silently logs to console as final fallback. Never let error capture throw.
**Warning signs:** Browser tab freezes, infinite mutation calls in Convex dashboard.

### Pitfall 4: Sensitive Data Leaking into devErrors
**What goes wrong:** Backend function args containing passwords, tokens, or API keys get stored in the devErrors table.
**Why it happens:** Naive serialization of function args without sanitization.
**How to avoid:** Create a `sanitizeArgs` helper that strips known sensitive field patterns (password, token, secret, key, authorization) before storage. Test with known sensitive field names.
**Warning signs:** Passwords visible in `/dev/errors` expanded details.

### Pitfall 5: Telemetry File Corruption on Concurrent CLI Commands
**What goes wrong:** Two `feather` commands run simultaneously, both read `.feather/telemetry.json`, both append, one overwrites the other's write.
**Why it happens:** No file locking on the JSON file.
**How to avoid:** Use append-only NDJSON (one JSON object per line) instead of a JSON array. Each write appends a line — no read-modify-write cycle. `feather stats` reads and parses all lines.
**Warning signs:** Missing telemetry events, corrupted JSON parse errors.

### Pitfall 6: devErrors Table Growing Unbounded
**What goes wrong:** In a dev environment with frequent errors, the devErrors table grows indefinitely, slowing queries.
**Why it happens:** No TTL or auto-cleanup.
**How to avoid:** Add a scheduled cleanup cron that deletes errors older than 30 days. Or limit to last 1000 errors (delete oldest when threshold exceeded). The `clearAll` button is manual; automatic cleanup prevents forgotten tables from growing.
**Warning signs:** `/dev/errors` page loads slowly, Convex storage usage spikes.

### Convex HTTP Action for Receiving Digest
```typescript
// Source: https://docs.convex.dev/functions/http-actions
import { httpAction } from "./_generated/server";
import { internal } from "./_generated/api";

export const receiveDigest = httpAction(async (ctx, request) => {
  // Validate shared secret
  const authHeader = request.headers.get("Authorization");
  if (authHeader !== `Bearer ${process.env.DIGEST_SECRET}`) {
    return new Response("Unauthorized", { status: 401 });
  }

  const digest = await request.json();
  await ctx.runMutation(internal.errorDigests.mutations.store, digest);

  return new Response(null, { status: 200 });
});
```

### Custom ErrorBoundary Component (Layer 1 — No Sentry)
```typescript
// When Sentry is NOT configured, use a custom ErrorBoundary for UI fallback
import { Component, type ErrorInfo, type ReactNode } from "react";

interface Props { children: ReactNode; fallback?: ReactNode; }
interface State { hasError: boolean; error: Error | null; }

export class ErrorBoundary extends Component<Props, State> {
  state: State = { hasError: false, error: null };

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    // Store to devErrors via mutation
    captureError("frontend", error, {
      componentStack: errorInfo.componentStack,
    });
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? <DefaultErrorFallback error={this.state.error} />;
    }
    return this.props.children;
  }
}
```

### CLI Telemetry NDJSON Pattern (Concurrent-Safe)
```typescript
// Append-only NDJSON — one JSON object per line, no read-modify-write
import * as fs from "node:fs";

export function appendEvent(filePath: string, event: TelemetryEvent): void {
  const dir = path.dirname(filePath);
  if (!fs.existsSync(dir)) fs.mkdirSync(dir, { recursive: true });
  fs.appendFileSync(filePath, JSON.stringify(event) + "\n");
}

export function readEvents(filePath: string): TelemetryEvent[] {
  if (!fs.existsSync(filePath)) return [];
  return fs.readFileSync(filePath, "utf-8")
    .trim()
    .split("\n")
    .filter(Boolean)
    .map((line) => JSON.parse(line));
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Class-based ErrorBoundary | React 19 `onUncaughtError`/`onCaughtError` hooks at `createRoot` | React 19 (2024) | Root-level capture without class components; still need class ErrorBoundary for UI fallback |
| `@sentry/browser` separate from `@sentry/react` | Unified `@sentry/react` includes browser + React-specific integrations | Sentry SDK v7+ | Single package for React apps |
| Static Convex cron only | `@get-convex/crons` component for runtime-configurable crons | 2024 | Allows schedule changes without redeployment — but adds complexity |

**Deprecated/outdated:**
- `componentDidCatch` alone for error capture: React 19 error hooks are now preferred for root-level capture. `componentDidCatch` still works for UI fallback.
- Sentry `Sentry.ErrorBoundary` as sole capture mechanism: For React 19, use `Sentry.reactErrorHandler()` with `createRoot` hooks for comprehensive capture; ErrorBoundary is supplementary for UI fallback.

## Open Questions

1. **Runtime-configurable cron schedule vs deploy-time**
   - What we know: Convex native `cronJobs()` is deploy-time. `@get-convex/crons` component provides runtime configuration.
   - What's unclear: Whether schedule changes are frequent enough to justify the added complexity of runtime crons.
   - Recommendation: Start with deploy-time crons (simpler). The crons.ts file is generated from feather.yaml config at deploy time. If users need runtime changes, migrate to `@get-convex/crons` later.

2. **Backend error wrapper implementation**
   - What we know: D-06 specifies backend errors should auto-capture. The project uses plain `mutation`/`query` from Convex.
   - What's unclear: Whether to create a custom wrapper (`errorCapturingMutation`, `errorCapturingQuery`) or use try/catch in each function.
   - Recommendation: Create wrapper functions (`withErrorCapture(mutation(...))`) that catch errors, store to devErrors, then re-throw. This is non-invasive — existing functions opt in by wrapping, new generated functions include it by default.

3. **Startup error sync**
   - What we know: D-06 says startup errors go to `.feather/errors.json` because backend may not be running.
   - What's unclear: Whether these should be synced to devErrors once backend becomes available.
   - Recommendation: Yes, sync on first successful backend connection. `feather stats` can read both local errors and Convex devErrors.
