# Phase 03.2.1.1: Create feather.yaml specs — Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Upgrade all feature YAML specs to complete, aspirational `feather.yaml` contracts. Scope covers all 8 entities: 5 CalmDo features (tasks, projects, subtasks, work-logs, activity-logs) + 3 example apps (todos, tickets, contacts). Each spec becomes the single source of truth for what the feature should do when fully realized. Old `.gen.yaml` and `.resolved.yaml` files are merged into `feather.yaml` then deleted.

</domain>

<decisions>
## Implementation Decisions

### Spec Completeness Standard
- **D-01:** Full parity with `tasks/feather.yaml` — every applicable section: fields, identity, behaviors, access, views, filteredViews, relationships, detailView, derivedData, hooks, indexes.
- **D-02:** Sections the generator doesn't consume yet (detailView, derivedData, hooks, overrides) are marked with `# TODO: generator pending` comments. They're part of the contract but not yet wired.
- **D-03:** `tasks/feather.yaml` is the gold standard reference — all other specs measured against it.

### Reality vs Aspiration
- **D-04:** Specs describe the **desired end state**, not current implementation. If subtasks should have promotion hooks, spec them. If projects should have status transitions, spec them.
- **D-05:** Gaps between spec and code are intentional — they represent the generator's backlog. When generators catch up (Phase 999.4), they read the spec and produce the ideal implementation.
- **D-06:** Ideal behaviors are specced even when not yet built: subtask promotion, project status transitions, work-log audit trail.

### Scope (Expanded from Original Phase Goal)
- **D-07:** All 8 entities get complete `feather.yaml` specs:
  - CalmDo features: tasks, projects, subtasks, work-logs, activity-logs
  - Example apps: todos, tickets, contacts
- **D-08:** Tasks included for reconciliation — merge `tasks.gen.yaml` content into `feather.yaml`, ensure no information lost.

### Old gen.yaml Cleanup
- **D-09:** For each entity with both `.gen.yaml` and `feather.yaml`: diff the two, merge any differences into `feather.yaml`, then delete `.gen.yaml`.
- **D-10:** Also delete `.resolved.yaml` files (activity-logs has one).
- **D-11:** After cleanup, `feather.yaml` is the single spec format across the codebase. No `.gen.yaml` files remain.

### Activity Logs Special Case
- **D-12:** Activity logs get a full `feather.yaml` with a top-level `kind: system` flag (or `systemGenerated: true`). Generator produces schema + read-only timeline view, skips CRUD forms/mutations.
- **D-13:** `actor` field upgraded from plain `string` to `belongs_to` relationship: `actor: { type: belongs_to, target: users, column: actorId }`. Consistent with how assigneeId works in tasks.
- **D-14:** Activity logs remain polymorphic via `entityType` + `entityId` fields. No standalone backend directory — logging stays inline in parent mutations.

### Claude's Discretion
- Field-level details for todos, tickets, contacts example apps — infer from existing `.gen.yaml` content
- Index optimization — add indexes that make sense for query patterns
- `kind: system` vs `systemGenerated: true` naming — pick what reads best in YAML

</decisions>

## Existing Code Insights

### Reusable Assets
- `tasks/feather.yaml` — Complete reference spec covering all sections
- `templates/defaults.yaml` — Shared defaults that apply to all feature YAML files
- Root `feather.yaml` — Project manifest listing all features

### Established Patterns
- Per-feature YAML at `src/features/{name}/feather.yaml` (999.1-D-01)
- Three-file chain: `defaults.yaml` → `feather.yaml` → `resolved.yaml` (03.2-D-06)
- Relationship types: `belongs_to`, `has_many`, `children`, `embedded` (03.2-D-17 through D-20)
- Auto-indexes from conventions: `by_userId`, `by_{relation}`, filterable fields (03.2-D-21)

### Integration Points
- Root `feather.yaml` `features:` array — must list all entities
- `convex/schema.ts` — table definitions must match feather.yaml field specs
- Current Plop generators read `.gen.yaml` — deleting these means generators won't work until 999.4 wires `feather.yaml` reading (acceptable trade-off per user decision)
