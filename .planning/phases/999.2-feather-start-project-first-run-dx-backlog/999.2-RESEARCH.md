## 2. `npm create` Package Pattern

### How `npm create <X>` Works

`npm create feather my-project` is equivalent to `npx create-feather my-project`. npm resolves `create-<X>` as the package name.

**Requirements:**
1. Publish a separate `create-feather` npm package
2. Package has a `bin` field pointing to the CLI entry
3. Entry script handles: arg parsing, git clone, branding, auth config, install

**Well-known examples:**
- `create-vite` — prompts for template, clones template dir from monorepo
- `create-next-app` — prompts for features, downloads template tar
- `create-react-app` — downloads template, installs deps
- `create-svelte` — interactive prompts, scaffolds from templates

### Package Structure for `create-feather`

```
create-feather/
  package.json          # name: "create-feather", bin: { "create-feather": "./dist/index.js" }
  src/
    index.ts            # Entry point — prompts + clone + scaffold
    prompts.ts          # Interactive wizard (3 questions)
    clone.ts            # Git clone base branch
    branding.ts         # Apply branding (from setup.ts)
    auth-config.ts      # Modify auth.ts based on selections
    scaffold.ts         # Run feather add for selected examples
```

### Decision: Where To Build

**Option A:** Separate repo/package (`create-feather`) published to npm.
- Pro: Clean separation, standard npm create pattern
- Con: Two repos to maintain, version drift

**Option B:** Inside this repo as a `packages/create-feather/` workspace.
- Pro: Single repo, shared code, easy testing
- Con: Monorepo overhead

**Option C:** Inside this repo as `scripts/create/` (not published yet).
- Pro: Simplest, iterate fast, publish later
- Con: Not immediately usable via `npm create`

**Recommendation:** Option C for Phase 999.2. Build the functionality as scripts that work locally (tested via `node scripts/create/index.ts my-project`). Phase 999.6 (Bundle Distribution) handles npm publishing. The curl fallback (`create.sh`) already exists for external users.

## 3. Base Branch Strategy

### The Hybrid Approach (D-12 from CONTEXT)

1. Maintain a `base` branch that has auth + shared infra + UI + test setup but NO feature code
2. `npm create feather` clones base branch, then scaffolds selected features via `feather add`
3. CI auto-generates base from main by stripping feature code

### What Needs Stripping for `base` Branch

**Remove (feature code):**
- `src/features/tasks/`, `src/features/projects/`, `src/features/subtasks/`, `src/features/work-logs/`, `src/features/activity-logs/`
- `src/features/todos/`, `src/features/tickets/`, `src/features/contacts/`
- `convex/tasks/`, `convex/projects/`, `convex/subtasks/`, `convex/work-logs/`, `convex/activity-logs/`
- `convex/todos/`, `convex/tickets/`, `convex/contacts/`
- Feature routes from `src/routes/_app/_auth/dashboard/`
- Feature nav entries from `src/shared/nav.ts`
- Feature schemas from `convex/schema.ts`
- Feature error constants from `src/shared/errors.ts`
- Feature i18n namespaces from `src/i18n.ts`
- Feature locale files from `public/locales/`
- Feature shared schemas from `src/shared/schemas/` (keep username.ts)

**Keep (infrastructure):**
- `src/features/auth/`, `src/features/dashboard/`, `src/features/onboarding/`, `src/features/settings/`, `src/features/uploads/`
- `convex/auth.ts`, `convex/auth.config.ts`, `convex/otp/`, `convex/password/`, `convex/devEmails/`, `convex/users/`
- All shared UI components (`src/ui/`)
- Test setup, Plop generators, templates
- Build config, tailwind, vite, tsconfig
- `bin/feather.ts` and commands
- `scripts/setup.ts` (basis for create script)

### CI Job Design

GitHub Actions workflow that:
1. Triggers on push to `main`
2. Creates a temporary worktree
3. Runs a strip script that removes feature code and cleans wiring points
4. Force-pushes to `base` branch
5. Runs tests to verify base still works

**Key challenge:** The strip script must cleanly modify `convex/schema.ts`, `src/shared/nav.ts`, `src/i18n.ts`, `src/shared/errors.ts` — removing only feature entries while keeping infrastructure entries. This is non-trivial string manipulation but well-suited to a Node.js script using AST or marker-based regions.

## 4. `feather add` Command Design

### Purpose
Install a feature (or example app) into an existing Feather project after creation.

### Approach
`feather add todos` would:
1. Read the feature's `feather.yaml` spec
2. Run the existing generator pipeline to scaffold files
3. Wire into schema, nav, i18n, errors, routes
4. Run `npx convex dev --once` to sync schema

**Important insight:** The existing `gen:feature` pipeline already does most of this. `feather add` is essentially a wrapper around the generator that takes a pre-defined spec instead of interactive prompts.

### Approach for Example Apps
Example apps already have `feather.yaml` specs. `feather add todos` reads `feather.yaml` from a known location (bundled with the base branch as `templates/examples/todos.feather.yaml` or similar).

## 5. Auth Provisioning Strategy

### Provider Selection Flow

```
Wizard Q2: Which auth providers?
  [x] Password (always enabled, zero-config)
  [ ] OTP (email codes)
  [ ] GitHub OAuth

If OTP selected:
  → Auto-enabled for dev (dev mailbox handles it)
  → "Resend API key needed for production — add RESEND_API_KEY to Convex dashboard"

If GitHub selected:
  → Prompt: GitHub OAuth Client ID?
  → Prompt: GitHub OAuth Client Secret?
  → Write to .env.local (or Convex env vars)
  → If skipped: disable provider, show instructions to enable later
```

### Auth File Modification

`convex/auth.ts` needs conditional provider inclusion. Current file imports all 3 providers. The create script would:

**Approach A (template-based):** Have multiple auth.ts templates:
- `auth-password-only.ts` — just Password
- `auth-password-otp.ts` — Password + ResendOTP
- `auth-all.ts` — Password + ResendOTP + GitHub

**Approach B (AST modification):** Keep current auth.ts, programmatically remove unwanted providers.

**Recommendation:** Approach A is simpler and more reliable. 3 templates is manageable. AST modification is overkill for a file this small (17 lines).

## 6. Post-Creation Verification

After project creation:
1. `npm install` — install dependencies
2. `npx convex dev --once` — initialize Convex backend, generate types
3. Restore `convex/tsconfig.json` (Convex CLI overwrites it — existing pattern in setup.ts)
4. Run `npm test` to verify everything works
5. Display success message with next steps

## 7. Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Base branch drift from main | CI auto-generation on every push to main |
| Strip script breaks wiring files | Integration test: generate base, run tests |
| `feather add` duplicates generator logic | Reuse existing Plop pipeline, don't duplicate |
| Auth template approach limits flexibility | 3 templates covers all current combinations; migrate to AST later if needed |
| Example app dependencies (CalmDo has cross-entity) | CalmDo is a bundle, not individual features for the wizard |
| npm publish logistics | Defer to Phase 999.6; use local scripts for now |
