# Phase 999.5: Excel-to-System Pipeline тАФ Research

**Researched:** 2026-03-30
**Phase Goal:** User uploads Excel file, system infers schema, user confirms, system produces feather.yaml + seed data

## Executive Summary

This phase builds the complete Excel-to-System pipeline: upload, LLM-powered schema inference, interactive confirmation wizard, feather.yaml generation, and data import with reconciliation for re-imports. Research covers technology choices, architecture decisions, and integration points with existing codebase.

## 1. Technology Choices

### 1.1 Excel Parsing: SheetJS (xlsx)

**Decision:** Use `xlsx` (SheetJS Community Edition) for Excel parsing.

- **Package:** `xlsx` on npm (Community Edition, Apache 2.0)
- **Capabilities:** Reads .xlsx, .xls, .csv, .ods; handles multi-sheet workbooks; extracts cell types (string, number, date, boolean); supports streaming for large files
- **Why not alternatives:**
  - `exceljs`: Better streaming but heavier (30MB vs 2MB), slower for read-only use
  - `read-excel-file`: Too minimal тАФ no multi-sheet, no type detection
  - `xlsx-populate`: Good but mainly for writing, not reading
- **Key APIs:**
  - `XLSX.read(data, { type: 'array' })` тАФ parse from ArrayBuffer (works in browser + Convex action)
  - `XLSX.utils.sheet_to_json(sheet, { header: 1, raw: true })` тАФ get raw typed data with headers
  - `sheet['!ref']` тАФ get data range; `sheet['!cols']` тАФ column widths (useful for display)
  - `workbook.SheetNames` тАФ list all sheets

**Integration with Convex:** SheetJS runs in any JS environment. For the web path, parse client-side (no upload needed for schema inference). For the CLI path, parse in Node. For re-import reconciliation, store parsed schema metadata in Convex (not the raw file).

### 1.2 LLM for Schema Inference

**Decision:** Use Claude via Convex `action` with `@anthropic-ai/sdk` for schema inference.

- **Why LLM (not heuristic-only):** Heuristics can detect basic types (number, date, email via regex). But entity classification (lookup vs master vs transaction), relationship detection, and semantic grouping require understanding of domain context. The LLM sees "Department, Employee Name, Hire Date, Salary" and classifies: Department = lookup entity, the rest = Employee master entity.
- **Where it runs:** Convex `action` (not query/mutation тАФ actions can call external APIs). Uses `ANTHROPIC_API_KEY` env var in Convex.
- **Model choice:** Claude Sonnet for speed + cost. Schema inference is a structured extraction task, not creative тАФ Sonnet handles it well.
- **Prompt structure:** Send column names + 3-5 sample rows + field type hints (from SheetJS cell types) + feather.yaml field type enum. Ask for: entity classification, field types, relationships, confidence scores.
- **Fallback:** If no API key configured, fall back to heuristic-only inference (covers 70-80% of cases). LLM is an enhancement, not a hard dependency.

### 1.3 Fuzzy String Matching: `fastest-levenshtein` + Custom Jaro-Winkler

**Decision:** Implement Jaro-Winkler in-house (~50 lines), use `fastest-levenshtein` as backup metric.

- **Why Jaro-Winkler:** Better than Levenshtein for name-like strings where common prefixes matter ("Emp Name" vs "Employee Name" = 0.87 Jaro-Winkler vs 0.38 normalized Levenshtein). The architecture doc specifies Jaro-Winkler with 50% weight.
- **Why implement vs library:** The `jaro-winkler` npm packages are either unmaintained or heavy. The algorithm is simple (50 lines), well-specified, and needs no dependencies.
- **Combined scoring (from architecture doc):**
  - Jaro-Winkler string similarity: 50% weight
  - Position similarity (index distance / total columns): 20% weight
  - Data fingerprinting (sample value overlap): 30% weight

## 2. Architecture

### 2.1 Pipeline Stages

```
[1. Upload/Parse] тЖТ [2. LLM Analyze] тЖТ [3. Wizard Confirm] тЖТ [4. Generate YAML] тЖТ [5. Import Data]
```

Each stage is independent and resumable. State persists in Convex tables between stages.

### 2.2 Convex Tables

**New tables required:**

1. **`imports`** тАФ Import history (D-12: first-class entity with permalink)
   - `id`, `userId`, `fileName`, `fileStorageId`, `status` (draft/analyzing/confirming/generating/importing/complete/failed)

   - `analysisResult` (JSON: LLM output тАФ entities, fields, relationships, confidence)

2. **`schemaMappings`** тАФ Persistent column mappings for re-import (D-02, D-09)
   - `id`, `userId`, `entityName`, `systemFieldId`, `systemFieldName`
   - `excelColumnName`, `excelColumnPosition`, `excelSheetName`
   - `importHistory` (array: date, oldName, newName records)
   - `dataFingerprint` (hash of sample values for reconciliation matching)

3. **`importErrors`** тАФ Per-row import errors (D-14, D-15)
   - `importId`, `entityName`, `rowNumber`, `severity` (green/yellow/red)
   - `column`, `originalValue`, `fixedValue`, `errorMessage`

## 3. Key Technical Challenges

### 3.1 Client-Side vs Server-Side Parsing

**Decision:** Parse Excel client-side for the web path.

- **Why:** Avoids uploading potentially large files just for schema preview. The user sees entities/fields instantly.
- **How:** SheetJS works in the browser. Parse the file client-side, extract schema + 5 sample rows per sheet, send just the metadata to Convex for LLM analysis.
- **For re-import reconciliation:** Must compare against stored mappings in Convex, so metadata is sent server-side.
- **For data import (stage 5):** Full file uploaded to Convex storage, parsed server-side in action for row-by-row import.

### 3.2 LLM Analysis Prompt Engineering

The LLM must return structured JSON matching the feather.yaml field types. Key prompt elements:

1. **System prompt:** "You are a schema analyst. Given Excel column headers and sample data, classify each column into feather.yaml field types."
2. **Available types:** string, text, number, boolean, enum, date, reference, email, url, currency, percentage (from `fieldTypeSchema`)
3. **Entity classification rules:**
   - Low cardinality (< 20 unique values) + short strings = enum or lookup entity
   - Name + contact info = master entity (people/orgs)
   - Name + description = reference entity (products/categories)
   - Dates + references + amounts = transaction entity
4. **Output format:** JSON matching FeatureYaml structure with confidence scores per field
5. **Sample prompt input:**

```json
{
  "sheets": [
    {
      "name": "Employees",
      "columns": ["EmpID", "Name", "Department", "Hire Date", "Salary", "Email"],
      "sampleRows": [
        ["E001", "John Smith", "Engineering", "2023-01-15", 85000, "john@company.com"],
        ["E002", "Jane Doe", "Marketing", "2022-06-01", 92000, "jane@company.com"],
        ["E003", "Bob Wilson", "Engineering", "2024-03-10", 78000, "bob@company.com"]
      ],
      "rowCount": 150,
      "detectedTypes": ["string", "string", "string", "date", "number", "string"]
    }
  ]
}
```

### 3.3 Reconciliation Algorithm

For re-import detection (D-09), the system needs to match a new upload against existing imports:

1. **Sheet name matching:** Compare new sheet names against existing entity names (Jaro-Winkler)
2. **Column pattern matching:** For each matched sheet, compare column names against stored mappings
3. **Data fingerprinting:** Hash first 50 rows of key columns, compare against stored fingerprints
4. **Threshold:** Combined similarity > 80% = auto-suggest as re-import. 60-80% = ask user. < 60% = treat as new.

### 3.4 Dependency-Order Import

When importing data across multiple entities with relationships:

1. Build dependency graph from relationships (references between entities)
2. Topological sort: import lookup entities first, then masters, then transactions
3. Track ID mappings: Excel row тЖТ Convex document ID (needed for reference resolution)
4. Handle circular dependencies: rare but possible тАФ break cycle with placeholder IDs, update after all entities imported

### 3.5 Large File Handling (Deferred but Designed For)

Architecture supports future chunked import:
- SheetJS streaming mode for files > 50MB
- Batch mutations (100 rows per mutation call) for data import
- Progress tracking via `importStats` field on import document
- UI shows progress bar during import stage

## 5. Validation Architecture

### 5.1 Verification Scenarios

| Scenario | How to Verify |
|----------|---------------|
| Single-sheet Excel тЖТ single entity YAML | Upload test.xlsx with 1 sheet, verify feather.yaml output matches expected fields/types |
| Multi-sheet Excel тЖТ multiple entities with relationships | Upload test.xlsx with 3 sheets (departments, employees, orders), verify 3 YAMLs with correct references |
| Re-import detects column rename | Upload v1.xlsx, then v2.xlsx with renamed column, verify reconciliation suggests rename |
| LLM fallback to heuristics | Remove API key, upload Excel, verify type inference still works (basic types) |
| Import data in dependency order | Upload with lookups + masters + transactions, verify lookups imported first |
| Error tiers displayed correctly | Upload with intentional bad data, verify green/yellow/red classification |
| Wizard state survives page refresh | Start wizard, refresh browser, verify wizard resumes at same step |

### 5.2 Test Data Sets

Need to create test Excel files:
1. **Simple:** Single sheet, 5 columns, 10 rows (happy path)
2. **Multi-entity:** 3 sheets with relationships (departments тЖТ employees тЖТ orders)
3. **Messy data:** Mixed types, empty rows, merged cells, inconsistent formatting
4. **Large columns:** 30+ columns requiring semantic grouping (D-07)
5. **Re-import:** Two versions of same file with renames, additions, removals

## 6. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| LLM returns invalid schema | Medium | High | Validate against featureYamlSchema, fallback to heuristics |
| SheetJS can't handle edge cases (merged cells, formulas) | Medium | Medium | Preprocess: unmerge cells, evaluate formulas to values |
| Reconciliation false positives (wrong rename match) | Low | Medium | Always require user confirmation, show confidence % |
| Large files cause browser tab crash | Low | High | Limit client-side parsing to first 1000 rows for preview |
| Convex action timeout on LLM call | Medium | Medium | Set timeout, retry with backoff, cache analysis results |
